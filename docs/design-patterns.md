# Design patterns — detail for patterns without a dedicated doc

`CLAUDE.md` carries a one-line index of every design pattern. The **detail** for each
pattern lives either in a dedicated `docs/<feature>.md` (see the pointer table below) or,
for patterns that don't warrant their own file, **in full here**. This keeps the
per-request `CLAUDE.md` small while keeping the rationale discoverable.

> **Documentation rule:** when you add or change behavior, update the detail (here or in
> the dedicated doc) **and** the one-line index entry in `CLAUDE.md` in the same change.
> Don't grow a dense multi-sentence bullet back into `CLAUDE.md`. Stale docs are a bug.

## Patterns documented here (no dedicated doc)

- **Lyrics (on-demand, plugin-sourced, editable)**: lyrics are fetched on demand from a **lyrics source plugin** and persist + can be user-edited. This added a **new `metadata` plugin kind** with a **`lyrics` capability** (`@nicotind/core` `plugin/{manifest,capabilities,index}.ts`: `LyricsCapability.fetchLyrics(LyricsQuery)→LyricsResult|null`; `validatePluginManifest` allows `lyrics` for `metadata` and **scopes the `defaultEnabled:true` ban to `acquisition` only** so a benign keyless source can default-on) — the documented "add a kind" recipe; the registry kernel is unchanged. First source: the **LRCLIB plugin** (`services/plugins/lrclib/`, pure-JS `fetch`, no key, returns both plain + synced LRC; `/api/get` exact match → `/api/search` fallback; injectable `fetchFn`), registered + `seedEnabled('lrclib',…)` in `index.ts` so it activates on first boot. **Transient failures are retried** (`request()` retries up to 3× with a short backoff on 429/5xx/network/timeout; a **404 short-circuits** as an authoritative no-match) — without this a throttled/timed-out *first* request surfaced as a false "no lyrics found" and the user had to click "Fetch lyrics" 2–3× before a warm request succeeded (`retryBackoffMs` is injectable so tests don't wait). Storage is the **`library_lyrics` side-table** keyed on the path-derived `songId` (`lyrics-store.ts`; `plain_text`/`synced_text`/`source`/`customized`/`updated_at`); plain text is **also written back to the file tag** (`audio-tags.ts` now writes/reads **ID3 USLT** + **Vorbis `LYRICS`**, like BPM) so it survives a move/transcode (synced LRC stays DB-only — no clean embedded standard). Routes (`routes/library.ts`, registry threaded via `LibraryRoutesOptions.pluginRegistry`): `GET …/lyrics` (any user → `LyricsDto`|null), `POST …/lyrics/fetch` (any user — keyless/benign; **returns a cached row unmodified**; on a DB miss it first **recovers lyrics from the file tag** (`source:'file-tag'` — the orphaned-row-after-transcode path), else queries `getEnabledWithCapability('lyrics')`, persists the first hit + writes the tag; **a `customized` user edit is never clobbered unless `force:true`**; `503` when no lyrics source enabled; **`502` when a source *errored* vs `null` for a confident no-match** — the route tracks whether any plugin threw so a transient failure surfaces as a retryable error instead of a false "no lyrics"), `PUT …/lyrics` (admin — sets `customized=1`, `source='user'`, **clears synced** since an edited body no longer matches its timing), `DELETE …/lyrics` (admin reset). Web: the `track-info-sheet` drawer's **Lyrics section** (`data-testid="lyrics-section"`, fetch/`edit`("Add" when empty)/save/reset) and the **now-playing lyrics panel** (`now-playing-lyrics-toggle` swaps the queue for a **karaoke-styled** panel that **highlights the active synced line** via the pure `lib/lrc-parser.ts` `parseLrc`/`findActiveLine` driven by `player.currentTime()`, with auto-scroll, fading-edge mask, and a scale-1.05 active line; falls back to scrollable plain text). The **empty state is a centered styled block** (`now-playing-lyrics-empty`) with a note icon and an inline **Fetch lyrics** button (`fetchLyricsManually()` force-fetches; `now-playing-lyrics-fetch`); a source failure (502) flips it to a distinct "Couldn't reach the lyrics source — Try again" via the `lyricsError` signal (vs a confident "No lyrics found"). A **fullscreen button** (`now-playing-karaoke-toggle`) expands the lyrics into an immersive gradient-covered overlay with glow orbs (palette extracted from cover art via `lib/cover-colors.ts`) and dedicated transport controls. **All lyrics surfaces are horizontal-overflow-safe**: every line/`<pre>` carries `break-words` and its scroll container is `overflow-x-hidden`. The fullscreen scroller adds `px-4` so the **active line's `scale-105` transform** paints into the padding instead of spilling past the viewport edge. Guarded by the `mobile-ux` e2e "long lyrics wrap and do not overflow horizontally". `LyricsDto` lives in `@nicotind/core` (+ re-export in `web/src/types/core.ts`). **Vocal mute (karaoke)** — *server-side ffmpeg, center-channel cancellation*: the fullscreen karaoke overlay's mic toggle flips `PlayerService.vocalsMuted`, and `player.component.ts` re-assigns `audio.src` through `ServerConfigService.streamUrl(id, token, { vocalsOff })`, appending `&vocals=off`. Server side, `streaming.ts` reads `?vocals=off` and **forces the transcode path even when `transcodeEnabled` is off** — filtering can only happen while re-encoding — as long as ffmpeg is present (no ffmpeg ⇒ graceful unfiltered passthrough, never a 500). `transcodeToFile` then applies `-af pan=stereo|c0=c0-c1|c1=c1-c0`: each output channel becomes the L−R difference, so anything mixed dead-center (typically the lead vocal) cancels while stereo-panned instruments survive. It's deterministic and dependency-free — imperfect (reverb/backing vocals leak; a mono downmix collapses toward silence), but this *is* the intended behaviour. `transcodeCacheKey` appends a `novox` marker **only** for the muted variant, so the filtered stream is its own cache entry while ordinary streams keep their existing keys (a deploy doesn't invalidate the whole transcode cache). Toggle state persists across tracks; position is preserved across the src reload via `restoredTime` (see [docs/web-ui.md](web-ui.md)). **History note:** this was gutted during the #134–#141 churn (the `vocalRemoval` param survived as a dead, eslint-disabled arg and the route never read `?vocals=off`), so vocal mute was a silent no-op until it was rewired — an earlier `stereotools` mid/side attempt was abandoned in favour of the `pan` cancellation above.
- **Guided acquire UX (raw network demoted)**: the catalog cards (album/EP/single, all routing to the same `huntCatalogAlbum` → `AlbumHuntModalComponent`) are the primary path; the raw network lane — peer bitrate/speed/queue stats, the Songs↔Folders toggle, and the full folder-tree `FolderBrowserComponent` — lives **only inside a collapsed "Advanced: browse Soulseek peers & folders" disclosure** (`data-testid="advanced-toggle"`), auto-opened only when there's no catalog hit so it stays the escape hatch, never the default (the acquirable hits surface in the blended Results list above it — see source-agnostic-acquisition.md). The album-hunt modal **leads with only the auto-selected best match** (`data-testid="hunt-best-match"`: "✓ Best match ready" + format/size/match%) and one Download action; the **full ranked peer list is opt-in** behind a collapsed **"Choose a different source (N found)"** disclosure (`data-testid="hunt-sources"`, holding the `hunt-candidate` rows + selection) so the common "just get it" case is a single tap, and its filters (FLAC/live/skew/min-match) are tucked behind a separate **"Explore more options"** `<details>` (`data-testid="hunt-options"`). A pasted/shared link is recognized in the same search omnibox as one more acquisition candidate — a **link-intent card** (chip auto-labelled YouTube/SoundCloud/Bandcamp/Spotify/Internet Archive/Link by `lib/link-intent.ts`, dispatched through `AcquireService.submit` — no backend choice exposed, no separate URL box or job list). → [docs/source-agnostic-acquisition.md](source-agnostic-acquisition.md)
- **Inline download lifecycle**: Search result cards show idle → blue progress wash + % → green "▶ Open in Library". Driven by `TransferService`; a `libraryDirty` signal tracks completed downloads (cleared on library page load, ignoring initial boot completions to prevent double-loading and disruptive auto-refreshes of active views). `TransferService` uses adaptive polling: 3 s when any transfer or acquire job is active, 30 s when idle. `kickPoll()` resets to fast mode immediately when a download is initiated. `AcquireService` mirrors the same first-poll guard via `hasRefreshed`: the initial `refresh()` (called by `LayoutComponent.ngOnInit`) silently baselines all existing terminal jobs without toasting; only subsequent running→done/failed transitions fire a toast — preventing the stale-replay flood of "added to library" messages on every app open.
- **Multi-user**: Shared music library (all users see all downloads). Per-user settings in bun:sqlite (`packages/api/src/db.ts`). First registered user becomes admin.
- **Service modes**: `embedded` (NicotinD spawns slskd as child process) or `external` (connects to a pre-existing slskd). Library/streaming stack is in-process — no music-server subprocess.
- **Auth flow**: NicotinD issues its own JWTs; holds auto-generated API key for slskd. Web player streams via `/api/stream/:id?token=` (JWT in query param). **Sliding sessions**: the JWT TTL defaults to **30 days** (`jwt.expiresIn`, `config/default.yml`) and the web silently renews it on every app boot — `provideAppInitializer` in `app.config.ts` fires `ApiService.refreshToken()` → `POST /api/auth/refresh` (`routes/auth.ts`, guarded by the existing `authMiddleware`, re-signs via `signJwt`) and swaps the token in via `AuthService.setToken()` (token-only, leaves username/role). So an active user effectively never hits the expiry wall and stops re-logging in daily; an actually-expired token 401s on refresh and the interceptor falls back to `/login`. **Share tokens are non-refreshable** (`/refresh` 403s on `share === true`) — they stay short-lived and read-only. The login page (`pages/login/`) is a fixed-palette glassmorphic card (indigo/fuchsia/teal aurora, theme-independent).
- **Route auth helpers + typed errors (`middleware/current-user.ts`)**: the repeated `const user = c.get('user'); if (user.role !== 'admin') return c.json({ error: 'Admin only' }, 403);` guard (21× in `routes/library.ts` alone) collapses to `requireAdmin(c)` — it returns the user and **throws a 403 `ForbiddenError`** (`@nicotind/core`, added alongside `AuthError`/`NotFoundError`/…) otherwise. `getCurrentUser(c)` is the typed one-call accessor for non-admin handlers. The central `errorHandler` maps any `NicotinDError` → `{ error, code }` at its `statusCode`; the library router now registers `app.onError(errorHandler)` on **itself** so a typed throw is mapped even when the router is mounted on a bare app (as the route tests do), not just under the global handler. This is the seed of routes-throw-typed-errors (vs. hand-rolled inline `c.json`); admin-denial bodies gained a `code` field (additive — clients key on the 403 status).
- **Artist page — tabbed, with a lazy Songs tab**: `artist-detail.component` presents **Albums | Singles & EPs | Songs** as a tab bar (`data-testid="artist-tab-{albums,singles,songs}"`, default = first populated tab) instead of stacked sections. The **Songs** tab is **lazy** (no fetch until first opened) and paginates via the library grid's IntersectionObserver pattern (`#songsSentinel`, `appendUnique`, `done`). It hits the new **`GET /api/library/artists/:id/songs`** (paginated; `sort`=newest|title|album via the pure `songOrderBy` whitelist; `starred=true` filter) and exposes a sort `<select>` + a Filters menu (`data-testid="artist-songs-filters"`). It supports a **strong multi-select with bulk actions** built on the existing `createSelection` (shift-click range) + an **extended `SelectionBarComponent`**: Play, Add to queue, Add to playlist, Download (offline) — each an **optional** output gated by a `can*` input (default off) so the playlist-detail usage is unchanged. (No Star bulk-action: the app has no star write-endpoint, so one wasn't invented.) **Admin delete parity**: because albumless files (loose songs that never grouped into an album) surface *only* on this Songs tab, it also offers deletion — a `canDelete` bulk button in the selection bar **and** a per-row destructive **Remove** action (`artistTrackActions`), both gated on `auth.role() === 'admin'` to match the hard backend gate on `DELETE /songs/:id` + `POST /songs/bulk-delete`. Removal prunes the local `songs` signal and records ids on `TransferService` (cross-view consistency), mirroring the album/genre-detail flow (`ConfirmDialogComponent` + `deleteError` banner). Without this the files were unremovable from any view.
- **Viewport-safe dropdown menus (`MenuPanelComponent`)**: filter/popover menus used a bare `absolute right-0` and could spill off a narrow viewport. The reusable `components/menu-panel/` wraps a projected `[menuTrigger]` + `[menuPanel]`, renders the panel `position: fixed`, and positions it with the pure **`computeMenuPosition`** (`lib/menu-position.ts`) — anchor-aligns under the trigger, **flips above** when there's no room below, and always **clamps fully into the viewport** (so even a panel wider than the space pins to the edge margin). The library **Filters** menu (`data-testid="library-filter-panel"`) and the artist **Songs** filter both use it; the pure math is unit-tested and an e2e asserts the panel's bounding box stays on-screen at 360px width.
- **Bottom-chrome stacking + full-screen scroll lock**: the mini-player and the mobile tab bar (`bottom-nav`) are both `z-50` — one plane — so a page-level modal backdrop (`z-50`, e.g. album-hunt) sits *under* both and they show/hide **together** (the old `z-40` bar got hidden by a modal while the player stayed visible). True full-screen sheets (Now Playing / add-to-playlist `z-[60]`, context menu `z-[70]`) still cover both. The Now Playing sheet pins the document via the ref-counted `ScrollLockService` (`services/scroll-lock.service.ts`: `overflow:hidden`+`overscroll-behavior:none` on `<html>` while ≥1 holder; nested overlays lock independently) so the backgrounded page can't scroll/overscroll behind it. The header's standalone download button was **removed** (Downloads is already a nav item) and its active-count badge folded onto the desktop "Downloads" nav link (`layout` `downloadCount()`; the mobile tab bar already carries it). The mini-player **grab "notch"** (`data-testid="player-grab"`) is now a solid full-opacity pill: it was invisible because its `bg-theme-muted` class **was never a registered utility** (only `bg-theme-{base,surface,surface-2,hover,accent,inverse}` existed), so it rendered transparent — `.bg-theme-muted` (→ `var(--theme-text-muted)`) is now registered in `styles.css`, which also fixes the settings status-dot that referenced the same class. Geometry/visibility contracts live in `lib/player-chrome.ts` + the `bottom-nav` docstring; covered by `bottom-nav`/`layout`/`scroll-lock` unit specs and the `mobile-ux` e2e (grab-notch visibility — the e2e screenshots the **static grab element**, not the playing mini-player, whose moving seek bar never stabilizes and would hang `screenshot()`).
- **Downloading albums suppressed from listing**: `GET /api/library/albums`, `GET /api/library/singles`, and `GET /api/library/artists/:id` exclude albums that are actively downloading (matched via `normalizeForGrouping`). Suppression keys on the **union** of (a) active `album_jobs` rows and (b) **in-flight slskd transfers** — the latter (pure `services/transfer-group-keys.ts` `transferGroupKeys`, parsing each active transfer dir's last two path segments → `albumGroupKey`, only for dirs with a non-completed file) covers **raw folder-browser / per-track grabs that never create a job row**, which used to surface mid-download (the Ultraviolence "5 copies showed while downloading" bug). The transfer fetch is cached ~4 s (`activeTransferKeys`, module-level, reset for tests via `__resetDownloadSuppressionCache`) so listing requests don't hammer slskd, and degrades to album_jobs-only when slskd is absent/unreachable. The paginated endpoints (`/albums`, `/singles`) apply the exclusion **inside the SQL `WHERE` (`downloadingExclusion()` in `routes/library.ts`, an `id NOT IN (…)` fragment), not as a post-`LIMIT` JS filter** — filtering after `LIMIT/OFFSET` shrank each page below its requested size, so the web's offset-accumulating grid (`library.component.ts`) re-fetched already-shown rows and rendered the same album 2–3× (and tripped `done` early, truncating the grid). `downloadingExclusion()` has a zero-cost fast path when nothing is downloading. The card reappears once every copy clears `active`/in-flight (slskd times out stuck transfers; the user can also cancel). `DownloadWatcher.start()` seeds `knownCompleted` from `completed_downloads` at boot so container restarts don't replay history.

# Library Scanner, Streaming & Cover Art

## Native library scanner

`LibraryScanner` (`packages/api/src/services/library-scanner.ts`) replaces the old `NavidromeSyncer`. `scanFull()` walks the music dir and reads tags via `music-metadata` (loaded through the optional-dep `music-metadata-loader.ts`, degrading to path inference if absent); `scanPaths()` does an incremental scan of a just-organized batch. It writes `library_albums`/`library_songs`/`library_artists`/`library_genres` directly.

**Incremental tag cache (skip re-parsing unchanged files)**: tag reads are the dominant scan cost, and a naïve `scanFull` re-parses the entire library on every boot. `readTracks` now stats each file, then partitions against the `scan_cache` table (`scan-cache.ts`): a file whose `size` **and** `mtime` match its cached row reuses the stored **raw** `ScannedTrack` (no `parseFile`); the rest are tag-parsed with bounded concurrency (`mapPool`, ~core count) and written back to the cache. Crucially the cache stores the _raw_ IO-derived tags, **not** the resolved `library_songs` values — so `resolveTags`/`buildLibrary` still run identically each scan and metadata overrides (keyed on the raw album id) keep surviving rescans. Any on-disk edit bumps `mtime`, forcing a re-parse. Result: an unchanged restart parses ~zero files.

**Database performance (`db.ts`)**: `applyPerformancePragmas` sets `synchronous=NORMAL` (safe under WAL, much faster for the big scan-write transaction), a ~64 MiB `cache_size`, a 256 MiB `mmap_size`, and `busy_timeout=5000` (wait out the background scan's writer instead of erroring). The default `/albums` grid — `WHERE hidden = 0 AND classification IN (...) ORDER BY created DESC` — is covered by a composite `idx_library_albums_grid (hidden, classification, created DESC)` so the filter + sort come from one index. **Download-suppression cache**: `downloadingExclusion` (`routes/library.ts`) only scans `library_albums` when a download is active; that scan is now memoized per-db by `albumIdsByGroupKey` (a short-TTL `WeakMap`, cleared via `__resetDownloadSuppressionCache`), turning a per-request full-table scan + normalization into an O(active-keys) lookup.

**Deterministic IDs**: `songId = sha1(relPath)`, `albumId = sha1(albumGroupKey(artist, album))`, `artistId = sha1(normalizeArtistForGrouping(artist))`. Artist IDs use `normalizeArtistForGrouping` (diacritics + case + whitespace only — punctuation preserved) so "Miranda!" and "Miranda" remain distinct; album IDs use `normalizeForGrouping` on the title (also strips punctuation so `¡Bang!…` variants collapse to one card). Edition and punctuation-variant folders collapse to one album at scan time — the duplicate-card merge is inherent to grouping, not a post-hoc reconciliation.

**Landing gate (`landed_at`)**: `library_songs.landed_at` (epoch ms, NULL = quarantined) gates whether a scanned song is _visible_ in the library. The scanner deliberately **omits `landed_at` from both the INSERT column list and the `ON CONFLICT … DO UPDATE SET`** in `persist()` — so a fresh insert mints NULL (quarantined until its required processing steps finish) and a rescan of an already-landed song preserves its timestamp. The windowed processor (`graduatePending`) is the **sole** writer that sets a timestamp; do not add `landed_at` to the scanner. A one-time, marker-gated backfill in `applySchema` (`library_sync_state` key `landing_backfill_v1`) lands every pre-existing row exactly once so an upgrade never retroactively hides music, and never re-runs (which would wrongly land in-flight downloads). Full mechanics: [library-processing.md](library-processing.md) "Landing gate"; listing enforcement: [download-pipeline.md](download-pipeline.md) "Process-before-landing".

Curation columns (`hidden`/`classification`/`manual_override`/`starred`) are keyed on the stable id and preserved across rescans; a full scan prunes rows whose paths no longer exist on disk. The whole flow is synchronous from the caller's view, so the former async-scan races are gone. `classification` is the **release type** (`album`/`ep`/`single`/`compilation`/`unknown`) — set by `LibraryCurator` metadata-first (Lidarr `albumType` from the `library_release_meta` side table, keyed on `albumId` like `library_artwork`) with a track-count heuristic fallback. See [download-pipeline.md](download-pipeline.md#release-type-model--albums-eps--singles-spotify-style).

**Manual full rescan (`POST /api/library/sync`)**: the `runSync()` route is admin-only and rebuilds the canonical `library_*` tables from disk (the same path the daemon uses on boot). It's surfaced in the web UI by the **"Sync library"** button in the Admin → System panel (`AdminComponent.syncLibrary()` → `LibraryApiService.resyncLibrary()`), which disables while in flight, toasts the outcome, and refreshes the scan-status count on success. Handy when files were dropped into the music dir out-of-band (no download trigger), or to force the prune of stale rows.

**Loose singles**: `resolveTags` calls the exported pure `isLooseSinglesBucket(dir, album)` — a track with no usable album (`Unknown Album`) or one in the synthetic `<Artist>/Singles/` bucket gets `album = title`, so each loose track mints its own single album (`albumIdFor(artist, title)`) and surfaces as its own card rather than collapsing into a hidden per-artist bucket. Format-dups of the same single still merge via the shared title group key.

**Metadata overrides (user-confirmed corrections)**: `resolveTags` also consults a `library_metadata_overrides` map (loaded by `scanFull`/`scanPaths`, threaded through `buildLibrary`/`selectLibraryTracks` as a pure `overrides` parameter). After deriving the raw artist/album from tags, it computes the raw `albumIdFor(artist, album)`, looks up an override, and substitutes the corrected `artist`/`album`/`year` **before** the artistId/albumId are minted — so a fix re-buckets the album under the correct artist. The key is the _raw_ (tag-derived) id, which the scanner reproduces every scan (tags never change), so corrections survive full rescans without moving files. See [metadata-optimize.md](metadata-optimize.md) "User-driven fix".

**Clean tracklist (one best file per track)**: before building rows, `buildLibrary` runs every album's files through `selectAlbumTracks` (`library-track-select.ts`) so the _consumed_ library is always one best-quality file per track regardless of on-disk mess — Soulseek folders routinely accumulate flac + mp3 + m4a + wav copies of the same songs, plus foreign/mislabeled rips.

- **With a canonical Lidarr tracklist** (from `album_jobs`, mapped by `albumIdFor` in `canonicalByAlbum()`): each file is keyed to the canonical track it matches (diacritic-insensitive `titlesOverlap`), the best format wins (`formatQuality` lossless > lossy, bitrate breaks ties), and files matching **no** canonical track are **dropped** ("as Lidarr proposes" — foreign tracks don't pollute the album).
- **Without one**: files collapse by normalized title (format-dups merge; nothing dropped as foreign).

Non-destructive: unselected files stay on disk but get no `library_songs` row, so a full scan's prune makes them invisible. Physical cleanup is `scripts/repair-album-folders.ts`. Incremental `scanPaths` selects within its batch; the full scan is authoritative.

**Album card de-duplication (inherent to the scanner)**: the pure helpers in `album-grouping.ts` (`normalizeForGrouping` strips diacritics, punctuation, a curated set of edition qualifiers — remaster/deluxe/anniversary/expanded/`(2 CD)`/trailing disc number — and standalone 4-digit years like `(2014)`) feed `albumIdFor(artist, album) = sha1(albumGroupKey(...))`. Every edition/punctuation-variant folder of one release resolves to the **same album id**; songs aggregate onto one `library_albums` row. Display name is the shortest member title (base edition wins over "(Deluxe Edition)"); `song_count`/`duration` are recomputed. Genuinely distinct titles ("Greatest Hits" vs "Greatest Hits II") stay separate — only curated edition keywords are removed, never bare words/numbers.

**VA / compilation handling (album artist vs track artist)**: `resolveTags` returns both `albumArtist` (for album grouping and `library_artists` ownership) and `trackArtist` (for `library_songs.artist`). When `albumArtist` matches VA patterns (`isVariousArtists` from `compilation-tagger.ts` — "Various Artists", "VA", "V.A.", etc.), the per-track `artist` tag is used as `trackArtist` so individual performers are preserved. `library_songs` stores both via `artist`/`artist_id` (track-level) and `album_artist`/`album_artist_id` (album-level) columns. `library_artists` rows are only created from album ownership (the accumulator in `buildLibrary`), so track-only artists on compilations don't pollute the artist list.

**Compilation detection** (`compilation-tagger.ts`): `classifyFolder` uses first-match-wins heuristics — COMPILATION flag or VA `albumArtist` in tags → compilation; coherent album + single artist → leave-alone; single-artist consensus → single-artist; folder name matches VA patterns → compilation; coherent album + ≥3 distinct artists → compilation; ≥5 distinct artists → compilation; large untagged dump → compilation. The "coherent album + multi-artist" rule is the key fix for well-tagged VA compilations (all tracks share one album name but have different artists). Distinct-artist counting is exact-string (lowercased only, no fuzzy/prefix matching), so it trusts whatever `artist` value each track resolved to — see the `inferMetadataFromPath` note below for why that value must already be clean.

**Track-number-prefix stripping in path inference** (`path-inference.ts`, `inferMetadataFromPath`): when ID3 tags are missing and a filename only splits into two `-`-delimited parts (e.g. `"01. Los Tekis - Escondiditos.mp3"`), the first part is normally used as-is for `artist`. A permissive regex strips a leading track-number token (`"01. "`, `"12) "`, `"5_"`, …) from that first part before it's assigned, mirroring the strip already applied to `title`. Without this, each track in an untagged folder mints a distinct garbled "artist" (`"01. Los Tekis"`, `"02. Los Tekis"`, …), which both pollutes the artist list with zero-album ghost rows _and_ trips the ≥3/≥5-distinct-artists compilation heuristic above, silently reclassifying a single-artist album as VA (breaking the `album_jobs` → `library_albums` `albumId` match used by the downloads-feed deep-link, since that id is a pure function of `artist`+`album` — see [download-pipeline.md](download-pipeline.md#unified-active-feed-downloaditem)).

**Compilations in the UI**: The main `/albums` grid excludes compilations (`classification = 'album'` only). Compilations have a dedicated `GET /api/library/compilations` endpoint and a "Compilations" tab in the library view. "Various Artists" is hidden from the `/artists` list. Artist pages show an "Appears On" tab listing compilation albums where the artist has tracks (`GET /api/library/artists/:id/appears-on`).

## Search matching (tokenized + diacritic-insensitive)

The user-reported trigger — searching **"C. Tangana Ídolo"** returned individual **Songs** but no album card, even though the release is a clean, visible `library_albums` row (`classification='album'`, `hidden=0`) — was primarily a **search-matching** bug, not fragmentation or classification. `LibrarySearchProvider.search()` used to pass the _entire query as one `LIKE '%…%'`_ substring against each column, relying on `COLLATE NOCASE`. Two failures fell out of that (both **verified against the prod DB**):

- **Multi-token queries never matched.** "C. Tangana Ídolo" is not a substring of the artist (`C. Tangana`) _or_ the title (`Ídolo`) alone, and no column holds both → **0 album hits**. Typing the artist and album together (the natural query) always failed.
- **Accents were significant.** `COLLATE NOCASE` folds case but **not diacritics**, so "Idolo" (no accent) missed "Ídolo" and instead matched an unrelated "Idolo De Multitudes".

**Fix:** matching moved into JS (`tokenize` / `matchesAllTokens`, now in the shared `services/search-tokens.ts`). The query is **folded** (NFD → strip combining marks → lowercase, so "Ídolo"→"idolo", "niño"→"nino") and split into tokens; a row matches only when **every** token is a substring of its folded haystack (`name + artist` for albums, `title + artist + album` for songs). SQLite still does the cheap `hidden`/`landed_at` gating; the JS scan over the visible rows (a few thousand) is a few ms. This makes "C. Tangana Ídolo", "idolo tangana" (reordered), and "idolo" (un-accented) all resolve to the right album — matters for an accent-heavy (Spanish) library. The `/search` **"In your library"** album section (below) is what renders those hits.

**Catalog lane shares the same matcher (the "La bifurcada" fix).** `CatalogService.search` (`catalog-search.service.ts`) scopes album cards to a matched artist when their own releases appear, but a **pure title search** (no artist matched) previously fell through to Lidarr's raw fuzzy `album.lookup` unfiltered. Lidarr ranks anything whose title _fuzzily_ matches, so a multi-word query with a rare second word ("La bifurcada") collapsed to the common first token and flooded the ALBUMS grid with unrelated "La"/"Là" albums — the second word looked _stripped_ even though nothing in our code splits the query. Fix: the fall-through now runs `filterAlbumsByRelevance(allAlbums, query)`, keeping only albums whose `title + artist` contains **every** query token (same folded, AND-semantics matcher extracted to `search-tokens.ts` and reused by `library-provider.ts`). Junk drops out; the network/folder acquisition lane still carries anything filtered. Unit-tested pure (`filterAlbumsByRelevance`) + as a `search()` regression.

## Fragmentation diagnostic

A **separate**, less common integrity problem (the search fix above already handles the reported Ídolo case): the same logical release split into multiple `library_albums` rows, or a full album that the grid hides. The detector (`services/library-fragments.ts`, `checkFragments`) is **calibrated against real prod data** so it stays a short, actionable list rather than a false-positive firehose — on the ~3.7k-album prod library it reports **4** clusters and **18** hidden rows (an earlier title-only / any-non-`album` design reported 23 mostly-false clusters and 1906 legitimate singles/EPs).

**Search-page surface — "In your library" album section.** The web's `/search` page used to render album cards only from the Lidarr `catalog()` lane (hidden for the **listener** role); `LibrarySearchProvider`'s `local.albums[]` was bound to nothing. `search.component` now owns a `libraryAlbums` signal populated from `res.local?.albums` with an "In your library" section above Songs, visible to every role. Click → `/library/albums/:id`. The provider selects `song_count` + `classification` (LIMIT 10→20) so EPs/singles render with a type chip + track count and quarantined rows are excluded.

**Detector 1 — same release, artist-spelling variants (`detectDuplicateAlbums`).** The scanner already auto-merges rows whose artist _normalizes identically_ (`albumGroupKey` uses `normalizeArtistForGrouping`, which strips diacritics/case but **preserves punctuation/spacing**), so the only fragments that survive are punctuation/spacing variants: "La Konga" / "La K'onga", "Mr Gato" / "Mr. Gato". Detection groups by `normalizeForGrouping(title)`, **sub-clusters by an alnum-only artist fold** (`artistFold`), and flags a sub-cluster only when its rows share that fold but differ in raw spelling or artist id. Sub-clustering by folded artist is what keeps genuinely-different same-title albums apart — "Off the Wall" (Michael Jackson vs Pink Floyd), "Greatest Hits" (six artists) — which the old title-only grouping wrongly flagged. **Fix:** alias the spellings via Admin → Library → Artist identity ("merge into"), then rescan (`POST /api/library/sync`).

**Detector 2 — full album wrongly hidden from the grid (`detectHiddenByClassification`).** The grid's `classification = 'album'` filter drops singles/EPs _by design_, so flagging every non-`album` row is noise. A real defect is a row whose track count contradicts its class — a `single` with >2 tracks or an `ep` with >7 tracks reads as a full album the grid wrongly hides (e.g. "Future Nostalgia" tagged `single`, 18 tracks). Also flagged: `classification = 'unknown'` and `hidden = 1` rows (both low-volume). Legitimately-short singles/EPs and compilations are **not** flagged. Each finding carries `songCount` so the UI shows the tell ("18 tracks, tagged single"). **Fix:** reclassify or unhide via the album row menu.

**Combined detector:** `checkFragments(db)` runs both detectors (plus the orthogonal one-track-per-title `checkMisSplitAlbums` from `library-audit.ts`) into one report shape consumed by:

- the **API endpoint** `GET /api/library/fragments` (admin-only — surfaced in the Admin panel as "Check fragmentation"),
- the **CLI script** `scripts/check-fragments.ts` (read-only; exit code 1 when any defect exists, so a scheduled run can alert),
- and unit-tested with the in-memory sqlite (`library-fragments.test.ts`).

## Multi-genre support

**One song, many genres.** File tags are multi-valued in practice — multiple ID3 genre frames and `;`/`,`/`|`-joined strings ("Alternative Country;Alternative Pop;…"; a prod dry-run found 1,355/9,791 tagged songs with semicolon lists). The scanner keeps the FULL set: `ScannedTrack.genre` holds every frame (old single-string scan-cache rows remain valid input), `buildLibrary` runs each value through the pure `splitGenres` (`genre-split.ts`), writes the ordered set to **`library_song_genres`** (`song_id, genre, position`; position 0 = primary) and mirrors the primary into `library_songs.genre` for zero-breakage single-value reads. `library_genres` counts a song under **every** genre it has.

**splitGenres rules (deterministic, unit-tested):**

- `;` `,` `|` always split; whitespace collapsed; case-insensitive de-dupe preserving order.
- `&` **never** splits (R&B, Drum & Bass, Melodic House & Techno).
- `/` splits **only when every side is a known genre** (library vocabulary ∪ aliases ∪ alias canonicals) — "Nu Disco / Disco" and "Pop/Rock" split; "Deep House / Vinyl" stays whole for the alias table.
- Display casing = the alias canonical, else the most common casing in the library (`buildKnownFromRaw` in-batch vocabulary merged under the db-loaded context so incremental scans behave like full scans).

**Genre aliases (`library_genre_aliases`, human-gated).** Everything the rules can't fix safely goes through propose→review→apply, exactly like artist aliases: `scripts/reclassify-genres.ts --propose` emits reviewable suggestions (junk drops like "Other"→'' — the genre goes NULL and the Lidarr enrichment refills it; punctuation/casing variant merges; **segmentations of no-separator concatenations** like "RockPunk"/"LatinWorld", see below; known-side extraction from unresolved `/` joins), `--apply` writes the rows, `--backfill` applies them to the stored sets immediately (else the next rescan does). Canonical may be a `;`-joined **list** (one alias → many genres) or `''` (drop). Side table — survives rescans; applied by `splitGenres` in `buildLibrary`, so messy files never need retagging.

**Concatenation segmentation (`segmentConcatenatedGenre`).** Mashed tags like `"LatinWorld"` or `"Tech-HouseDowntempoTechnoMinimal TechnoHouse"` match nothing but other identically-mis-tagged tracks, which starves radio's candidate pool — the root cause of the José Larralde incoherence (see [radio.md](radio.md) → "Case study"). The splitter walks the value **longest-known-segment first with backtracking**, so multi-word genres win over their own prefixes ("Soft Rock", not "Soft"), and matches the vocabulary **punctuation/space-insensitively** so a `Tech-House` fragment lands on the library's `Tech House`. Two rules keep it safe enough to propose:

- **Cuts only at an uppercase letter directly preceded by a letter or digit.** A space or hyphen protects a compound, so `Pop Rock` (781 songs), `Dance-Pop` (433) and `Singer-Songwriter` (237) can never be torn apart — this guard was added _after_ a dry run against the real vocabulary proposed exactly those splits.
- **Every segment must resolve to a known genre** (all-or-nothing, mirroring the `/` rule) and be ≥3 characters, so `BritPop`, `Neo-Psychedelia`, `Moogsploitation` and `RnBSwing` produce no proposal.

Still propose-only: `splitGenres` never segments at scan time, it only expands reviewed alias rows. Against the real 657-genre vocabulary this yields 10 proposals covering ~200 songs, with zero false positives.

**Backfill without a rescan (`backfillGenresFromAliases`).** A newly-applied alias would otherwise wait for a full rescan. The backfill re-runs `splitGenres` over each song's _stored_ set and `setSongGenres` on any that changed — idempotent, and rescan-safe because `buildLibrary` applies the same aliases anyway. This is the one genre write that legitimately **replaces** a primary instead of appending to it (the append-only rule below exists to protect _detected_ genres from clobbering each other; a human-reviewed alias re-minting `"LatinWorld"` → Latin/World is exactly the intended override).

**Scan-cache flush (one-time).** Pre-multi-genre cache rows stored only the first genre frame, so `applySchema` bumps a `scan_cache_version` marker and flushes `scan_cache` once — the next scan re-parses all files and recovers multi-frame extras. Never re-runs.

**Out-of-scan writes.** `setSongGenres(db, songId, genres)` (`genre-split.ts`) is the single mutation path outside the scanner (genre enrichment task, admin `POST /songs/:id/genre` — which accepts a `;`-joined list, `backfill-genre.ts`): replaces the join rows, mirrors the primary, refreshes the touched `library_genres` counts. File-tag writes always write the full set joined `"; "`. Consumers: filters match via `EXISTS` on the join table, radio scores `genreSetCloseness` over the sets (see [radio.md](radio.md)), recipe `where` fragments are rewritten by `expandGenreWhere` (curated-playlists.ts) so `s.genre LIKE …` sees every genre, and `GET /songs/:id` ships `genres: string[]` for the track-info sheet's chips.

**Licence / rights** (`library_songs.licence` + `licence_source`): the scanner reads a canonical rights code from the file's `LICENSE`/`COPYRIGHT`/`WCOP` frames via `licenceFromTags` (`audio-tags.ts`, normalised by `@nicotind/core` `normalizeLicence` — positive IDs only, never guesses ARR from a bare © notice), threaded through `ScannedTrack → SongRow → persist` with `licence = COALESCE(excluded.licence, library_songs.licence)` — same durability as `bpm`/`genre`. The scanner never writes `licence_source`, so a manual `user` source survives rescans. Fills where `licence IS NULL` come from the background `licence` enrichment task (tags → MusicBrainz) or a curator; NULL = unknown. The album-level aggregate `library_albums.licence` is the **unanimous** track code (`unanimousLicence`; else NULL) computed in the `buildLibrary` reduce — so an album reads as "Public Domain" only when every track is PD — surfaced on the `Album` DTO and filterable directly. Full design in [music-licence.md](music-licence.md).

**Multi-artist support**: Songs and albums can have multiple artist associations via join tables (`library_song_artists`, `library_album_artists`). Each link carries a `role` (`primary` or `featuring`) and a `position` for ordering. The parser (`artist-split.ts`) extracts featuring credits (`feat.`, `ft.`, `featuring`, `with`, `w/` — always, unconditionally) and detects primary-artist delimiters (`;`, `&`, `, `, `/`, `+`, `and`, `x`, `y`, `con`, `vs`). Semicolon is a known acquisition divider (it mirrors `genre-split`'s separators, and Soulseek/URL sources join co-artists with it) — like every delimiter it only splits when **every** part is a confirmed artist, so a stray `;` never mangles a name. Dash is intentionally **not** a delimiter (a real-library scan found only junk like `BUR - BUR`).

**Confirmation-gated splitting (the "works out of the box" fix).** Splitting is **conservative**: `splitArtists(raw, { confirmedArtists, canonicalWhole })` splits a compound into individual artists **only when every part is a confirmed real artist**, and otherwise keeps the string whole — so a band/duo name is never mangled. The old guard was the inverse and _self-defeating_: it added _every_ raw artist string (including the compound ones themselves) to a `knownArtists` set and kept a string whole only if the whole string was "known" — but a compound is always self-referentially present, so nothing ever split (verified against the real library: 105/10,476 songs had >1 artist and 49 of those were garbage `Bob Marley` + `Bob Marley & The Wailers` prefix pairs). The two sets are assembled offline by `loadSplitAuthority` (`artist-identity-store.ts`):

- **`confirmedArtists`** — every artist string that appears **atomically** (no delimiter, no featuring — `isAtomicArtist`) anywhere in the library, ∪ resolved members from the authority table below. A compound can never confirm _itself_. `buildLibrary` also folds in atomic names from the current batch (so a full scan is self-contained), and the caller loads the whole-library set from the DB (fixing the old per-batch incremental inconsistency).
- **`canonicalWhole`** — compound strings Lidarr/MB says are **one act** (bands/duos like "Wisin & Yandel"); kept whole even when both members are independently confirmed.

**Cached authority (`library_artist_identity`).** So the _synchronous_ scanner never makes a network call, split decisions are cached in a side table (like `library_artwork`, keyed on `artistIdFor(rawCompound)`, survives rescans/prunes): `decision` ∈ `single` (→ `canonicalWhole`), `split` (→ members feed `confirmedArtists`), `unknown` (recorded to avoid re-query, contributes nothing — library-only logic decides). It's populated by the **`artist-identity` windowed enrichment task** (Lidarr `artist.lookup`; see [library-processing.md](library-processing.md)) and seedable up-front by `scripts/resolve-artist-identity.ts` (dry-run default, `--apply`). **With no Lidarr the table stays empty and splitting still works** via library-atomic confirmation alone (e.g. "Charly García y Luis Alberto Spinetta" splits when both appear solo in the library).

The existing `artist`/`artist_id` columns on songs/albums remain as the primary artist for backward compat; the join tables provide the full multi-artist picture. Artist-detail and songs queries use `OR id IN (SELECT ... FROM library_song_artists WHERE artist_id = ?)` to include multi-artist associations. The API attaches an `artists` array (`{ id, name, role }[]`) to song and album responses via `attachSongArtists`/`attachAlbumArtists` (batch helpers in `artist-attach.ts`) — wired into `/artists/:id`(+`/songs`,`/appears-on`), `/albums`, `/singles`, `/compilations`, `/albums/:id`, `/songs/:id`(+`/similar`), `/random`, `/recent-songs`, `/genres/songs`, playlist songs, **and the unified-search `library-provider`**. The frontend renders linked names via `ArtistLinksComponent` (clickable `routerLink` per name; `&` for primaries, `feat.` prefix for featuring), used by the shared `TrackRowComponent` (search/genre/playlist/artist/album pages) + the album/artist headers. (Whole-card album-grid tiles and the queue-row buttons keep a plain string on purpose — an inner `routerLink` inside an `<a>`/`<button>` would nest interactive elements.) Backfill is a full rescan (`POST /api/library/sync`), which repopulates the join tables through `buildLibrary → persist` and prunes the old garbage links.

**Acquisition→library artist cohesion (shipped).** Acquisition persists the canonical artist identity the moment a download is decided, so the scan that lands it already knows the artist — no background-window latency. `recordAcquiredArtistIdentity` (`artist-identity-store.ts`) writes (1) a `library_artist_identity` row `decision='single'`, `source='lidarr'` — the Lidarr canonical artist name is one act by definition, so a compound canonical like "Bob Marley & The Wailers" is instantly protected as `canonicalWhole` — and (2) the artist's MBID into `artist_discography_links` (keyed on the same deterministic `artistIdFor` id the scanner will mint; preserves any existing `lidarr_id`). Call sites, all best-effort try/catch beside the equivalent artwork write: the interactive `hunt-download` route (`routes/discography.ts`), and the shared unattended `acquireAlbum` core (`album-acquire.ts`, on the enqueued path) with the MBID threaded from the watchlist poller (`watchlist.artist_mbid`) and the auto-acquire sweep (Lidarr `artist.foreignArtistId`).

**Spelling-variant aliases (`library_artist_aliases`).** `artistIdFor` is a pure string hash, so "Snoop Dog" and "Snoop Dogg" mint two separate entities that never merge. The alias side table (`alias_norm` → `canonical_name`, survives rescans) fixes this _before_ ID minting: `buildLibrary` rewrites `trackArtist`/`albumArtist` **and each split-member credit** through `SplitAuthority.aliases` (loaded by `loadSplitAuthority`) before any `artistIdFor`/`albumIdFor` call, so variants collapse into one entity on the next rescan and the full-scan prune drops the orphaned rows. Alias pairs also feed the split-confirmation set (an MBID-matched variant is a confirmed real artist). Accepted trade-off: albums under an aliased-away spelling get new deterministic album IDs (their old per-album artwork overrides/stars are lost) — one-time, per merged variant.

**Alias derivation is human-gated — do not automate it.** `deriveMbidAliases` groups `artist_discography_links` by MBID and _proposes_ merging spellings that share one MusicBrainz id — but it deliberately does **not** run unattended (not in the `artist-identity` window task), because the MBID cache is only as trustworthy as its writer: `DiscographyService.upsertLink` stores Lidarr's **top fuzzy lookup hit** (`candidates[0]`) with no name verification. Verified against the real library: the only live MBID-equal pair was **"Âme" vs "ME"** — two different artists whose lookups fuzzy-matched to the same MusicBrainz id; an automatic merge would have corrupted the library. Flow: `scripts/resolve-artist-identity.ts --aliases` prints the proposals (dry run), a human reviews, `--apply --aliases` writes the reviewed set (`source='mbid'`); admin-driven merges write `source='user'` rows, which task/script writes never overwrite (`upsertArtistAlias` conflict guard).

**Split compounds hidden from the artist grid (`split_compound`).** After a collab splits, the compound entity still exists — `song.artist_id`/album grouping key the whole raw string for backward compat — which would leave a "Charly García y Luis Alberto Spinetta" tile in the grid _next to_ the two member tiles. `buildLibrary` flags every artist entity whose name resolves to >1 primary credit, `persist` writes it to the **scanner-owned** `library_artists.split_compound` column (deliberately separate from the user-curated `hidden`, which the scanner never touches), and `GET /api/library/artists` adds `split_compound = 0`. The row stays fully functional for direct navigation/search, member pages still list the collab album via the `library_album_artists` join, and the flag is recomputed every scan so it self-corrects when the split authority changes (e.g. a user later marks the compound one act). Zero-album artists with loose songs are untouched — they're real artists, not pollution (verified on the live DB: all 102 zero-album artists own ≥1 song).

**User-fixable splits, merges & renames (shipped).** Admins can correct any wrong identity decision; a user decision is the highest authority and permanent. `POST /api/library/artists/identity` (admin) accepts four shapes — `{rawName, decision:'single'}` (keep the compound as one act), `{rawName, decision:'split', members:[...]}` (force-split, ≥2 members), `{rawName, mergeInto}` (spelling-variant alias onto a **different** artist), and `{rawName, rename}` (fix this one artist's own spelling/name) — then runs the full rescan **synchronously** and returns `200` once the change has landed, so the UI shows a spinner and refreshes with the result instead of racing a background job (the old fire-and-forget `202` gave no feedback). Both `mergeInto` and `rename` reuse `library_artist_aliases` (`source='user'`), which `aliasFix` applies to every credit's display spelling **before** `artistIdFor` mints the id — the one difference is the guard: `mergeInto` rejects an equal-normalized target (that would be a no-op merge), while `rename` **allows** it, because an accent/case-only fix (e.g. "Los Áutenticos Decadentes" → "Los Auténticos Decadentes", both normalizing to the same id) is exactly the point — same id, corrected display. A `rename` whose normalized form differs mints a new id (a genuine rename, e.g. "[traditional]" → a real name). Precedence is enforced twice: `upsertArtistIdentity`/`upsertArtistAlias` conflict guards mean a background (lidarr/mb/library) write can never overwrite a `source='user'` row (only another user decision can), and `pendingArtistIdentityRows` excludes user rows regardless of TTL so the window task never re-resolves them (same contract as `library_metadata_overrides`: user confirmation is canonical, survives rescans). UX entry points, both admin-gated and both opening the shared `ArtistIdentityModalComponent` (radio _one act / separate artists / rename this artist / same as another artist_, member chips + rename field prefilled by the client-side `splitArtistParts` mirror of `splitOnDelimiters` — server re-validates): the artist page header's pencil button (`data-testid="artist-identity-fix"`; on success it routes to the artists grid, since a split hides the compound and a rename-to-new-id changes the page's id), and the track-info sheet's "fix" link next to the artist line (`data-testid="track-artist-identity-fix"`), which acts on the song's **raw tag artist string** and so covers compounds without their own artist page.

## Genre overrides (curator-correctable genres) — issue #187 task A3

**The problem.** Genre enrichment could only ever _append_ (`appendSongGenres`), so a wrong **primary** genre was unfixable: the primary is `library_song_genres` position 0, mirrored into `library_songs.genre`, and it is what the radio scorer leans on. José Larralde — Argentine folk — sat at primary `Latin`, so his radio surfaced Shakira and Enrique Iglesias.

**Why a side table and not a `genre_source` column.** The obvious shape (mirroring `licence_source`) does not work here. `persist` **deletes and rebuilds every rescanned song's `library_song_genres` rows from the file tags** ("Replace (not merge) each rescanned song's genre set"). `library_songs.genre` is COALESCE-preserved, but the join rows — the multi-genre set `genreSetCloseness` actually reads — are not. Under the old design, enrichment survived a rescan _only_ because it mirrored the merged set back into the file tag; if that tag write failed (read-only file, unsupported container, permissions) the enrichment was silently reverted. Overrides therefore live in a side table applied at scan time, like `library_genre_aliases`/`library_artist_aliases`. **This inverts the durability contract: the DB is authoritative and the file-tag mirror is now best-effort, not load-bearing.**

**`library_genre_overrides`** — `(scope, key)` primary key, where scope is `artist` | `album` | `song`:

| scope    | `key`                                 | why                                                          |
| -------- | ------------------------------------- | ------------------------------------------------------------ |
| `artist` | `normalizeArtistForGrouping(name)`    | matches `artistIdFor`'s input; move-proof                    |
| `album`  | `albumGroupKey(albumArtist, album)`   | edition-collapsing; matches `albumIdFor`'s input; move-proof |
| `song`   | the song id (`sha1("song:"+relPath)`) | **not** move-proof — the exception, not the default          |

Rows are keyed **at the granularity the source provides** (MusicBrainz gives genres per release-group, Lidarr per artist, a future Essentia head per track), so one row fixes every song it covers and is inherited automatically by tracks downloaded later. `genres` is a `;`-joined ordered list, primary first; `''` suppresses every genre (junk drop, same convention as a `library_genre_aliases` empty canonical).

**Merge semantics differ by source, and this is load-bearing.** `applyGenreOverride` (pure, `genre-overrides.ts`) resolves `song > album > artist` — the most specific scope wins outright, scopes are never merged with each other — then:

- **`source='user'` replaces the set outright.** `genreSetCloseness` is a position-blind **MAX** over every genre pair, so leaving a broad tag genre in place keeps scoring 1.00 against everything in that broad genre and completely masks the correction. Measured: overriding Larralde to `Folclore;Chacarera` while retaining his `Latin` tag still surfaced Enrique Iglesias and Babasónicos at genre 1.00. A curator who types an exact list means that list; "Reset to file tags" undoes it.
- **automated sources prepend and keep the rest**, correcting the primary without destroying what the file carried — the right trade-off when a machine picked the genres.

**Review queue is a `status` column (`applied` | `pending` | `rejected`), not a review file.** `reclassify-genres.ts` is the wrong precedent to copy: it grades a 66-row _vocabulary_ once. The closer precedent is `library_artist_identity` — per-entity human decisions carrying `source`, where `source='user'` is permanent and automation never overrides it (enforced in `upsertGenreOverride`). The decisive reason is that a file flow **has no memory of rejection**: re-running `--propose` over 2,474 artists would re-propose everything already declined. `--out`/`--apply <file>` still exist as transport for bulk editing; the table is the state.

**Applying without a full scan.** `backfillGenreOverrides` re-applies every applied override to the stored genre sets, the same role `backfillGenresFromAliases` plays for aliases — a full scan of a large library is minutes for what is otherwise an instant edit. Measured 464 ms across 14,469 songs. It computes keys with the real `albumGroupKey`/`normalizeArtistForGrouping` (a SQL `LOWER()`/`TRIM()` approximation would miss every diacritic case and silently no-op). The artist-genre route calls it before the synchronous rescan; a **reset** needs the rescan, since the backfill can only add an override's effect, never undo one.

**UI.** Artist page header, beside the identity pencil: `data-testid="artist-genre-fix"` opens `ArtistGenreModalComponent` (`requireCurator`). It shows the genres currently in effect **with their provenance** ("from file tags" / "set by you" / "from musicbrainz") — the cue that makes a wrong genre obvious — a `;`-separated editor with a live primary-first preview, and states plainly that the list replaces what is in the files. `POST`/`DELETE /api/library/artists/:id/genre` write `source='user'`, `status='applied'`, run the rescan synchronously (same choice as the identity route: immediate feedback), and `recordAudit`. Song scope is reachable via `POST /api/library/songs/:id/genre` with `mode: 'replace'` (default stays `append`, so existing callers are unchanged). **Album-level UI is deferred** — the data model and `resolve-genres.ts` support album scope already.

**A reconcile gap fixed alongside.** `reconcileAlbums`'s `buildLibrary` call passed neither the genre context nor the overrides, so an album reconcile rebuilt genre sets from raw tags — reverting reviewed **aliases** as well as overrides for the touched albums until the next full scan. Both are now passed.

## Trusted-metadata genre (MusicBrainz) — issue #187 task A1

`scripts/resolve-genres.ts` (`--propose` / `--apply` / `--status` / `--reject` / `--reconsider` / `--revert`) resolves genres **by MBID only**, never fuzzy-by-name, writing proposals into `library_genre_overrides`. MBIDs are persisted first in `library_mbids` (`(scope, key)`, side table for the same prune-survival reason as above; `upsertMbid` keeps the better-sourced id so a fuzzy `mb-search` can never downgrade one read from a file tag).

**Resolution is album-first.** Measured on the prod library (2026-07-23, 25 sampled artists owning genre-less songs):

| source                               | artists resolved                                 | songs fixable                           |
| ------------------------------------ | ------------------------------------------------ | --------------------------------------- |
| Lidarr artist genres                 | 2 / 25                                           | 60 / 1,946 (3%)                         |
| MusicBrainz **artist** genres        | 2 / 25                                           | 60 (byte-identical — Lidarr proxies MB) |
| Spotify artist genres                | 0 — API requires a premium sub for the app owner | 0                                       |
| MusicBrainz **release-group** genres | 8 / 12                                           | 371 in that sample alone                |

So artist-level lookup buys ~3% and **Lidarr is not an independent source**; release-group level covers ~6× more and is far more specific (`chacarera`, `cumbia`, `progressive house` vs a flat `Latin`). Do not re-litigate artist-level-first — this is the same kind of recorded null result as the radio embedding weight.

**The confidence gate exists because of a real false pair.** An exact-name Lidarr lookup for "Emilia" (Argentine, 26 songs) returned the _Swedish_ Emilia, genre `hip hop`. Exact name equality is therefore not sufficient to auto-apply:

| resolution                                                                                       | confidence | status    |
| ------------------------------------------------------------------------------------------------ | ---------- | --------- |
| MBID from the file's own tags                                                                    | 1.0        | `applied` |
| album: artist **and** album title match                                                          | 0.8        | `applied` |
| artist: exact name **and** ≥1 library album title corroborated in the candidate's release groups | 0.8        | `applied` |
| exact name only, uncorroborated                                                                  | 0.5        | `pending` |
| fuzzy / top-hit                                                                                  | 0.3        | `pending` |

`verifyGenre`'s existing `hits[0]` fallback is exactly the uncorroborated path and must never feed an auto-applied override. **Empty MB genres produce no row at all** — not a `pending` row — or the queue fills with thousands of empty non-decisions.

**What A1 cannot do.** MusicBrainz has _no_ genres for José Larralde at artist level and only 3 of 25 release groups carry any. The Larralde case is closed by a **user** override, not by A1; A1's realistic yield on this library is a few hundred songs. The remaining lever, if that disappoints, is a folksonomy source (Last.fm / Discogs) with much better Latin coverage — deliberately not built, since #187 prefers curated genres over folksonomy tags.

## Native streaming + cover art

`streamingRoutes` (`packages/api/src/routes/streaming.ts`) serves `GET /api/stream/:id` straight from disk via `Bun.file` with HTTP `Range`/`206` support (path looked up from `library_songs.path`, traversal-guarded under the music root). Optional **ffmpeg transcoding** is gated by admin streaming settings (`streaming-settings.ts` in the `app_settings` table; `transcode.ts` spawns ffmpeg, probing availability once).

`GET /api/cover/:id` resolves a **manual artist override first** (see "Artist images" below), then **canonical artwork** (see below), then folder art (`cover.jpg`/`folder.jpg`/…), then embedded art, caching extracted/fetched images under `dataDir/cover-cache`. For an **album** id with no canonical art, the disk fallback picks the album's **first track** (`ORDER BY disc, track`) as the representative — _not_ an arbitrary `LIMIT 1` row — so the album shows track 1's folder cover (the real album art) rather than a wrong thumbnail from a mislabeled sibling file.

**Artist ids never fall back to a track's album art.** `resolvePath` has _no_ `artist_id` branch: an artist id (a distinct sha1 namespace, never matched by `album_id`) with no override and no canonical poster resolves to **404**, so `CoverArtComponent` shows the neutral initial-on-gradient tile. A representative track's cover is a _wrong_ face for an artist (often an old/misleading release) — worse than the clean placeholder.

Successful cover responses carry `Cache-Control: public, max-age=86400` so the browser caches them and navigation stops re-requesting every tile — the connection-pool pressure that otherwise stalled album pages. The remote canonical fetch (`fetchRemoteCover`) is bounded by `AbortSignal.timeout` (6 s) so a slow/dead Lidarr URL can't hang a request (and a browser connection slot); on timeout it falls through to on-disk art. Negative (artless) results are still short-circuited by the module-level `noArtCache` (10 min TTL).

`GET /api/cover/:id?embedded=1` is a special mode that serves **only** the file's embedded picture — skipping both canonical and folder art — for a _song_ id, cached under a distinct `<id>~emb` key. It backs the Fix-metadata cover picker (so a user can preview/choose the artwork baked into a specific track); extraction is the shared `extractEmbeddedPicture` (`services/cover-sources.ts`), which the normal `extractCover` also delegates to for its embedded fallback. See [metadata-optimize.md](metadata-optimize.md) "Cover picker".

The album/EP **detail track list omits the per-track thumbnail** (every row shares the album cover): `TrackRowComponent`'s `showCover` input is `false` there, cutting ~12–20 identical cover requests per page. Mixed-album lists keep it `true`.

**Sized thumbnails (the "covers load super slow" fix)**: the cover route used to serve the _full-resolution_ source image for every slot — a multi-MB album cover shipped to render a 40px player thumbnail — because it ignored the `size=` the web already sends. It now snaps `size=` to a small bucket set via `bucketCoverSize` (`services/cover-thumbnail.ts`, `{40,80,160,320,640}`; absent/over-max ⇒ original) and serves a resized **WebP** (`resizeCover`, lazy `import('sharp')`) cached at `<baseKey>@<size>.<ext>` in the shared `cover-cache` dir — so repeat thumbnail hits are one small file read, and a resize failure transparently falls back to the original. `purgeCanonicalCache` (`artwork-store.ts`) prefix-deletes every `c_<key>@<size>` variant too, so a corrected cover leaves no stale thumbnails. Frontend: `CoverArtComponent`'s `<img>` is `loading="lazy"` (`coverLoadingAttr`, override with the `eager` input) + `decoding="async"`.

**Transcoded streams are seekable**: on-the-fly transcoding used to be a sequential ffmpeg _pipe_ (status 200, no `Content-Length`/`Accept-Ranges`), so far seeks did nothing (the iOS/Firefox "seek to the end does nothing" bug — acute because the Opus standardization means iOS _needs_ transcoding to play at all). `transcode-cache.ts` (`getTranscodedFile`) now transcodes the whole file **once** to a disk cache (`<dataDir>/transcode-cache`, keyed on source path+**mtime**+format+bitrate so a re-encode invalidates it; concurrent plays de-duped via an in-flight map; oldest-first eviction over a 2 GiB soft budget), and **both** the transcode and passthrough paths serve through the shared `serveFileWithRange()` helper — so every stream advertises ranges and seeks correctly. `transcode.ts` `transcodeToFile()` writes atomically (temp + rename). First play warms the cache; replays are instant and seekable.

**Firefox "never plays" bug — `nativeAppCors()` was silently stripping `Content-Length` off every stream response.** `serveFileWithRange` always sets an explicit `content-length` header, but `hono/cors`'s built-in middleware appends its post-request `Vary: Origin` header via `c.header()` — which, once a route has already returned a `Response` (`Context#finalized`), rebuilds that `Response` from `res.body`. Reading `.body` on a `Bun.file()`/Blob-backed `Response` converts it into a generic `ReadableStream`, and Bun writes an unknown-length stream as `Transfer-Encoding: chunked`, dropping the declared `Content-Length` — for **every** `/api/stream` response, 200 or 206, verified live with `curl -D -`. Chrome's `<audio>` tolerates a chunked, length-less 206 range read; Firefox's does not, and gets stuck endlessly re-requesting/stalling — which the buffering-spinner UI (see [web-ui.md](web-ui.md) "Playback loading feedback") then faithfully renders as a spinner that never resolves and a track that never actually plays. Fixed by hand-rolling `nativeAppCors()` (`packages/api/src/middleware/cors.ts`) to mutate `c.res.headers` directly instead of calling `c.header()` after `next()`, so the Blob body — and its real `Content-Length` — reaches Bun untouched. Pinned by a wire-level regression test (`cors.test.ts`, real `Bun.serve` + `fetch()`, since Hono's in-process `app.request()` doesn't reproduce it) that fails on the old `hono/cors`-based implementation.

## Canonical artwork

Soulseek rips often carry missing/low-res/wrong embedded art; audio files carry no artist photo at all. Fix: the `library_artwork(id, kind, cover_url, updated_at)` table stores canonical URLs keyed on the **same deterministic ids the scanner mints** (`albumIdFor`/`artistIdFor`) — kept off the scanner-managed tables on purpose, so it survives full rescans/prunes untouched and can be written at hunt time _before_ the album is scanned onto disk.

`artwork-store.ts` exposes:

- `resolveArtwork` — direct album/artist hit, or song→album mapping so a per-track request (e.g. the player) resolves the album cover
- `setArtwork` — upsert + purge stale `c_<key>` cache when the URL changes
- Lidarr image pickers

The cover route prefers `resolveArtwork` → lazily fetches the remote URL into a `c_<key>` cache namespace → serves it; only if no URL exists or the fetch fails does it fall back to on-disk art.

The scanner sets album `coverArt = albumId` and artist `coverArt = artistId` (songs keep their own id) so requests key the store correctly.

**Population**:

1. `hunt-download` writes album + artist artwork from the Lidarr payload.
2. The **`artist-image` windowed enrichment task** auto-fills artist posters in the background — see [library-processing.md](library-processing.md). It's the standing population path; the script below remains for one-shot/manual runs.
3. `scripts/backfill-artwork.ts` (dry-run default, `--apply`) backfills the existing library by matching artists via `artist_discography_links`/name and albums via edition-stripped group key against the monitored Lidarr list (`artwork-backfill.ts`).

Two opt-in passes widen coverage when the artist isn't monitored:

- `--album-lookup`: targeted per-album `album.lookup("<artist> <album>")` for substantial albums (default >3 tracks, `--min-tracks N`) still missing art — skips Singles/Various-Artists junk.
- `--lookup-missing`: slow per-artist lookup for every non-monitored artist (pathological on a large library, hence off by default).

The web renders artist thumbnails via `CoverArtComponent` (gradient+initial fallback preserved) in the artists grid and artist-detail header.

### Artist images (auto + manual override)

Artist photos come from two layers, both keyed on `artistIdFor(name)` so they survive rescans:

- **Auto (canonical)** — `resolveArtistImageUrl` (`services/artist-image.ts`) resolves a real portrait **Lidarr poster first, Spotify portrait as fallback** (see [spotify-fallback.md](spotify-fallback.md)), shared by the manual backfill script and the windowed `artist-image` task. The result is a `library_artwork(kind='artist')` URL the cover route fetches.
- **Manual override** — a user (admin) can **upload** a photo or **copy one of the artist's album covers**, stored as _bytes_ in the persistent `<dataDir>/artist-overrides/<artistId>.<ext>` dir (`services/artist-image-override.ts`) — bytes because an upload has no URL and a disk-only album cover has no public one. Routes (`routes/library.ts`): `PUT /api/library/artists/:id/image` (multipart upload, JPEG/PNG/WebP ≤ 8 MB), `POST …/image/from-album` (`{ albumId }` → copies that album's resolved cover bytes), `DELETE …/image` (revert). An override sets `library_artists.manual_override = 1` so the auto task leaves the choice alone, and is served **ahead of** canonical artwork by the cover route. `DELETE` clears the flag → the artist reverts to auto/placeholder. The web edit affordance lives on the artist-detail portrait (`artist-image-*` testids), admin-gated.

## On-demand track analysis (BPM + genre)

The `track-info-sheet` drawer exposes an **Analysis** section that fills in per-track metadata the rip didn't carry.

**Reading stored values.** The sheet renders BPM/genre from its `effectiveSong()` — the caller's `Song` input when given, else one lazily fetched via `getSong(songId)` (`GET /api/library/songs/:id`) on init. This fetch matters because the player opens the sheet with only a `songId` and display strings (no `Song`); without it, already-stored `bpm`/`genre` (e.g. from the windowed enrichment) would render as "Unknown" even though the columns are populated.

**BPM.** `library_songs` has an additive `bpm INTEGER` column. The scanner reads it from tags (`music-metadata` `common.bpm`), and `persist()` updates it via `bpm = COALESCE(excluded.bpm, library_songs.bpm)` so a rescan of a file whose tags lack BPM never wipes an analyzed value. `POST /api/library/songs/:id/analyze`:

1. returns the existing tag value immediately (`source: 'tag'`) when present;
2. otherwise `track-analysis.ts` `analyzeBpm()` decodes a ~90 s mono PCM slice via ffmpeg (`-ar 44100 -ac 1 -f f32le`, the rate `music-tempo`'s onset detection expects) and runs `music-tempo` (lazy-imported, degrades to `null` when absent);
3. on a value it **writes the tag back** (`audio-tags.ts` now emits `TBPM` for ID3 and Vorbis `BPM` for FLAC/Opus/M4A) _and_ sets `library_songs.bpm`, so the result is durable across rescans.

**Genre.** `GET …/genre-suggestion` runs `verifyGenre()` — a Lidarr `artist.lookup` (diacritic/punctuation-insensitive name match via `normalizeForGrouping`, reading the artist's `genres`), returning `{ current, suggested, candidates, source }` and degrading to `source: null` when Lidarr is unconfigured or has nothing. Admin-only `POST …/genre` applies a chosen value: writes the file tag and updates `library_songs.genre`. The web gates the **Apply** button on `AuthService.role === 'admin'`.

All ffmpeg-dependent paths are guarded by `ffmpegAvailable()`. Response types (`BpmAnalysisResult`, `GenreSuggestion`) live in `@nicotind/core` (re-exported to the web via `packages/web/src/types/core.ts`).

### Bulk backfill scripts

The per-track buttons only touch one song at a time, so batch scripts fill the whole library — the same offline-vs-Lidarr split as `backfill-years.ts` (offline) vs `optimize-metadata.ts` (needs Lidarr). Both are **dry-run unless `--apply`**, write to the canonical column **and** the file tag (so a rescan keeps the value), append to `<dataDir>/*.log`, and **resume on re-run** (selection is "still missing", writes are incremental). The thin selection/grouping logic is the pure, unit-tested `services/track-backfill.ts` (`resolveSongAbsPath`, `groupSongsByArtist`, `planGenreBackfill`); the heavy lifting reuses the already-tested `analyzeBpm` / `verifyGenre`.

**`scripts/analyze-bpm.ts` — offline** (ffmpeg + music-tempo, no Lidarr). Walks every `bpm IS NULL` song; per song it prefers a BPM already on the file tag, else runs `analyzeBpm()`. Each `analyzeBpm` is a ~90 s ffmpeg decode, so work runs through a bounded worker pool (`--concurrency`, default 3); `--limit N` takes a test slice. Aborts early if ffmpeg isn't on PATH.

```bash
bun run packages/api/src/scripts/analyze-bpm.ts                 # dry run
bun run packages/api/src/scripts/analyze-bpm.ts --apply         # write DB + tags
bun run packages/api/src/scripts/analyze-bpm.ts --apply --limit 50 --concurrency 4
```

**`scripts/backfill-genre.ts` — needs a live Lidarr** (genre comes from the artist's Lidarr/MusicBrainz metadata). Walks songs with no genre, groups them by artist, and looks each artist up **once** (`verifyGenre`), fanning the suggested genre out to all of that artist's genre-less songs. It **replaces** the empty set with Lidarr's (`setSongGenres`). Lidarr config + `secrets.json` fallback mirror `optimize-metadata.ts`; exits non-zero if no API key. The `library_genres` facet counts are **not** recomputed here — they refresh on the next full scan (matching the per-song genre route).

```bash
bun run packages/api/src/scripts/backfill-genre.ts             # dry run
bun run packages/api/src/scripts/backfill-genre.ts --apply     # write DB + tags
```

**`scripts/append-genre-backfill.ts` — needs a live Lidarr**, the **append** counterpart: it runs over the **whole** library (not just empty-genre songs) and **unions** Lidarr's genres into each song's existing set via `appendSongGenres` (case-insensitive dedup → idempotent, safe to re-run), writing the merged set to the DB + file tag. Use it to enrich songs that already carry tag genres without clobbering them; the dry run previews only the songs that would actually gain a genre. `--limit N` bounds the pass.

```bash
bun run packages/api/src/scripts/append-genre-backfill.ts               # dry run
bun run packages/api/src/scripts/append-genre-backfill.ts --apply       # write DB + tags
bun run packages/api/src/scripts/append-genre-backfill.ts --apply --limit 500
```

# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Quality Gates

Every task on this project must satisfy all three gates before being considered done:

1. **Every change must be tested.** New features get new tests. Bug fixes get regression tests. Refactors must not reduce coverage. If a change can't reasonably be unit-tested, add an integration or e2e test instead — untested code is not shippable.

2. **Every test must run in CI.** Adding a test locally is not enough. Verify the relevant GitHub Actions workflow actually executes the new test on push. If a new test file or package is added, confirm it's picked up by `.github/workflows/`. Don't close out a task until CI covers the new test.

3. **Documentation must be updated in the same change as the code.** This is not optional and not a follow-up task: **every time you add or modify behavior, update the docs in the same commit/PR.** Significant decisions — new patterns, new services, why an approach was chosen over alternatives, trade-offs accepted — must be captured. If a change makes an existing doc statement wrong, fix that statement; stale docs are treated as a bug. Where docs live:
   - **Inline in `CLAUDE.md`** (the default) — add or update a bullet in the relevant section (Key Design Patterns / Web UI). This is the established pattern; most subfeatures are one dense bullet here.
   - **A dedicated `docs/<feature>.md` file** when a feature is large enough that an inline bullet can't do it justice. When you do this, **add a one-line pointer to it from `CLAUDE.md`** so the main file remains the index — a reader should never have to discover a `docs/` file by accident.
   - **A concise `// why` comment in code** for local rationale that belongs next to the implementation.

   A change is not "done" (gate-complete) until its documentation reflects reality.

## What is NicotinD?

NicotinD is a unified music acquisition + streaming platform that orchestrates **slskd** (Soulseek P2P client) behind a single API, web UI, and CLI, and **natively scans/streams** the music library itself (Navidrome was removed — see Architecture). Downloads from Soulseek land in a shared folder; the DownloadWatcher organizes and incrementally scans completed transfers into the canonical SQLite library that the API streams from. URL-based acquisition (yt-dlp / spotdl) feeds the same pipeline.

## Commit Conventions

This repo uses [Conventional Commits](https://www.conventionalcommits.org/). Every commit message must follow this format:

```
<type>(<optional scope>): <description>
```

**Types that bump the version** (appear in CHANGELOG):
| Type | When to use | Version bump |
|------|-------------|--------------|
| `feat` | New user-facing feature | minor |
| `fix` | Bug fix | patch |
| `perf` | Performance improvement | patch |

**Types that don't bump** (hidden from CHANGELOG):
| Type | When to use |
|------|-------------|
| `chore` | Deps, tooling, config, CI tweaks |
| `refactor` | Code restructuring (no behavior change) |
| `style` | Formatting, whitespace |
| `docs` | Documentation only |
| `test` | Adding/updating tests |
| `ci` | CI pipeline changes |
| `build` | Build system changes |

**Breaking changes**: Add `BREAKING CHANGE:` in the commit body or `!` after the type (e.g. `feat!: remove legacy auth`) to trigger a major bump.

**Enforcement**: A `commit-msg` hook via husky + commitlint rejects non-conforming messages.

**Releasing**: When ready to release, run `bun run release`. It reads the commit history since the last tag, determines the version bump, updates `package.json`, generates/updates `CHANGELOG.md`, commits, and creates a git tag.

## Architecture

```
NicotinD (Hono API :8484)  — native library scanner + streaming, all in-process
└── slskd (Soulseek client :5030)  ──── shared /data/music folder
        DownloadWatcher → LibraryOrganizer → LibraryScanner (tags → SQLite)
```

> **Navidrome was removed.** NicotinD is now fully native: it scans the music
> dir itself (`LibraryScanner`, `music-metadata`), serves audio bytes from disk
> with HTTP range support (optional ffmpeg transcoding), and resolves cover art
> from folder/embedded images. The canonical `library_*` SQLite tables are the
> single source of truth. The **`/rest/*` Subsonic proxy and the playlist feature
> were dropped** in the same migration (playlists to be re-added natively later).

**Bun monorepo** with workspace packages:

| Package                     | Purpose                                                                                                             |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `@nicotind/core`            | Shared types (Zod schemas), logger (pino), crypto utils, error classes                                              |
| `@nicotind/slskd-client`    | Typed HTTP client wrapping slskd's REST API (`/api/v0/*`)                                                           |
| `@nicotind/service-manager` | Strategy pattern for managing sub-service lifecycle (child_process or Docker)                                       |
| `@nicotind/api`             | Hono API server — routes, JWT auth, unified search, download watcher, native library scanner + streaming, SQLite DB |
| `@nicotind/web`             | Angular v22 web UI (standalone components, signals, Tailwind)                                                       |
| `@nicotind/cli`             | Commander.js CLI (Phase 3)                                                                                          |

**Entry point**: `src/main.ts` — loads config, starts services, wires clients into the API server.

## Key Design Patterns

- **Source-agnostic acquisition (the north star)**: acquisition/search is **not** Soulseek-first. Every acquirable result from any source (Soulseek, archive.org, Spotify, future) maps to one `AcquisitionCandidate` (`@nicotind/core`, with a discriminated `acquire` intent: `{via:'url'}` → `/api/acquire`, `{via:'enqueue',sourceRef,files}` → slskd) and the UI renders **one blended, ranked list with a neutral per-row source chip** (`app-source-chip`) + a single Get — no "the network"/"Advanced"/"From X"/"Also on…" hierarchy. Synchronous metadata sources blend server-side: `CandidateSearchAggregator` (`services/candidate-search.ts`, `GET /api/sources/search`) for search and `AlbumHuntOrchestrator` over `SourceHunter[]` (`services/source-hunter.ts`, `POST /api/discography/albums/:id/hunt/sources`) for hunt, both registry-gated (`isEnabled(id)`); Soulseek keeps its async/two-phase live search and is blended **client-side** (`lib/acquisition-candidate.ts` `mergeAndRank`, `songResultToCandidate`). The acquisition gate (`requireAcquisitionMiddleware`) checks `hasAnyAcquisitionEnabled()` (any source, not only `download`). **Adding a source = one adapter + a pure mapper, no route/UI change.** → See [docs/source-agnostic-acquisition.md](docs/source-agnostic-acquisition.md).
- **Native library scanner**: `LibraryScanner` (`packages/api/src/services/library-scanner.ts`) walks the music dir, reads tags via `music-metadata`, and writes to `library_*` SQLite tables with deterministic SHA1 IDs. Includes clean tracklist selection (one best file per track) via `selectAlbumTracks`, Lidarr-canonical matching, and edition-collapsing album IDs. `resolveTags` (threaded through `buildLibrary`/`selectLibraryTracks` as a pure `overrides` map, loaded from `library_metadata_overrides`) substitutes user-confirmed artist/album/year corrections **before** minting the artistId/albumId, so a fix re-buckets the album and survives full rescans (see the user-driven metadata fix bullet). → See [docs/library-scanner.md](docs/library-scanner.md).
- **Native streaming + cover art**: `streamingRoutes` serves `GET /api/stream/:id` from disk with HTTP Range/206 and optional ffmpeg transcoding; `GET /api/cover/:id` resolves canonical → folder → embedded art. 404 responses carry `Cache-Control: public, max-age=300` and a module-level `noArtCache` (10 min TTL) short-circuits the slow `extractCover()` IO on repeat misses. **Sized thumbnails (the "covers load super slow" fix)**: the route used to serve the *full-resolution* source image for every slot — a multi-MB album cover shipped to render a 40px player thumbnail — because it **ignored the `size=` the web already sends**. It now snaps `size=` to a small bucket set via `bucketCoverSize` (`services/cover-thumbnail.ts`, `{40,80,160,320,640}`; absent/over-max ⇒ original) and serves a resized **WebP** (`resizeCover`, lazy-`import('sharp')`) cached at `<baseKey>@<size>.<ext>` in the shared `cover-cache` dir — so repeat thumbnail hits are one small file read, and a resize failure transparently falls back to the original. `purgeCanonicalCache` (`artwork-store.ts`) now prefix-deletes every `c_<key>@<size>` variant too, so a corrected cover doesn't leave stale thumbnails. Frontend: `CoverArtComponent`'s `<img>` is `loading="lazy"` (`coverLoadingAttr`, override with the `eager` input) + `decoding="async"` so a grid of tiles doesn't fire dozens of eager, main-thread-decoded requests at once. **Single-album track lists drop the redundant per-track thumbnail**: every row in an album/EP detail page shares the *same* cover, so `TrackRowComponent` gained a `showCover` input (default `true`) and album-detail passes `[showCover]="false"` — the track number carries row identity, eliminating ~12–20 identical cover fetches/decodes per album page. Mixed-album lists (playlists, artist Songs, search, the queue) keep per-track covers. The cover route also serves an **embedded-only** image via `?embedded=1` (skips canonical+folder, `~emb` cache key) for the Fix-metadata cover picker — see the user-driven metadata fix bullet. **Transcoded streams are seekable**: on-the-fly transcoding used to be a sequential ffmpeg *pipe* (status 200, no `Content-Length`/`Accept-Ranges`), so far seeks did nothing (the iOS/Firefox "seek to the end does nothing" bug — acute because the Opus library standardization means iOS *needs* transcoding to play at all). `transcode-cache.ts` (`getTranscodedFile`) now transcodes the whole file **once** to a disk cache (`<dataDir>/transcode-cache`, keyed on source path+**mtime**+format+bitrate so a re-encode invalidates it; concurrent plays de-duped via an in-flight map; oldest-first eviction over a 2 GiB soft budget) and **both** the transcode path and the passthrough path serve through the shared `serveFileWithRange()` helper — so every stream advertises ranges and seeks correctly. `transcode.ts` `transcodeToFile()` writes atomically (temp + rename). First play of a track warms the cache (one transcode); replays are instant and seekable. → See [docs/library-scanner.md](docs/library-scanner.md).
- **Canonical artwork**: `library_artwork` table stores canonical URLs keyed on deterministic IDs (survives rescans); populated by `hunt-download` and `scripts/backfill-artwork.ts`. → See [docs/library-scanner.md](docs/library-scanner.md).
- **Metadata optimization (fix wrong/poor covers + year)**: `metadata-optimize.ts` re-fetches better cover/year/release-type from Lidarr and **overwrites** what's stored — distinct from `backfillArtwork`, which only fills *missing* art. `optimizeAlbum()` matches the global `album.lookup("<artist> <title>")` (normalized title+artist, reusing `looksLikeNonAlbum`/`normalizeName` from `artwork-backfill.ts`), then `setArtwork` (purges the stale cover cache), updates `library_albums.year` (placeholder `0001` dropped), and `setReleaseType`. This **automatic** matcher is all-or-nothing and intentionally conservative: it's the cheap **bulk** backfill (`optimizeAllAlbums`, default-scoped to albums missing artwork/year; `POST /api/admin/metadata-optimize?all=1&dryRun=1` + admin Settings button + `scripts/optimize-metadata.ts`). It **skips placeholder-artist albums** (`isPlaceholderArtist` in `artwork-backfill.ts` — `<Desconocido>`/`Unknown`/`Various Artists`/bracketed): the artist guard could never pass and the lookup would be poisoned, so those are deferred to the user-driven fix below. → See [docs/metadata-optimize.md](docs/metadata-optimize.md).
- **User-driven metadata fix (correct a wrong artist/album; confirm low-confidence)**: the all-or-nothing matcher fails when the *stored artist itself is wrong* — a rip tagged `<Desconocido>` searches `"<Desconocido> Selva"`, a **poisoned query** a known band never matches, leaving the wrong cover. `metadata-fix.ts` is the interactive answer: `searchCandidates()` runs an **editable** Lidarr query (default `"<artist> <album>"`, but **album-title-only when the stored artist is a placeholder** per `isPlaceholderArtist` — so `<Desconocido>` searches `"Selva"` and actually returns La Portuaria; the modal shows an amber hint prompting the user to type the real artist, and the box stays user-overridable) and `rankCandidates()` (pure, diacritic-folded `NFD` token overlap) returns the best-first top 8 **including low-confidence hits the user confirms**; `applyMetadataFix()` applies a confirmed candidate **or free-text** (`{artist?,album?,year?,coverUrl?,releaseType?}`). The correction persists in **`library_metadata_overrides`** (keyed on the scanner's **raw** `albumId`; `corrected_album_id` reverse-lookup via `findByCorrectedId` avoids orphaning a row on re-correction) which `resolveTags` consults at scan time, and the canonical tables are mutated **immediately** to match: songs are UPDATEd in place (`songId` is path-derived → `starred`/`hidden`/`playlist_songs` survive), the album row + `library_artwork`/`library_release_meta` are re-pointed to the corrected id, the corrected artist is upserted and the orphaned old artist pruned via the shared `pruneOrphanArtist` (`services/library-aggregates.ts`, also reused by album-delete). Admin-only: `GET /api/library/albums/:id/metadata-candidates?q=` (needs Lidarr) + `POST /api/library/albums/:id/metadata` (free-text works offline) → album-detail **"Fix metadata"** button opens `MetadataFixModalComponent` (`data-testid="optimize-metadata"`); on apply the page re-fetches the corrected album by its (possibly new) id and cache-busts the cover. **Cover picker (cover-only, multi-source)**: the modal's top **Cover section** (`data-testid="cover-picker"`) lets you change *just* the artwork without touching artist/album/year — the all-or-nothing candidate "Apply" only ever carried the top candidate's single cover, so a stale/wrong cover got stuck. `GET …/cover-candidates?q=` aggregates the **current** cover, **deduped Lidarr alternatives** (reuses `searchCandidates`; omitted, not 503, when Lidarr is down), and one entry per **distinct image embedded in the album's own tracks** (pure `services/cover-sources.ts` `selectDistinctEmbeddedCovers` over `hashBytes`-deduped `extractEmbeddedPicture`; thumbnails served by the cover route's new `?embedded=1` mode — embedded-only, skipping canonical+folder, cached under a `~emb` key). `POST …/cover` (`ApplyCoverRequest`) applies a chosen cover: a `coverUrl` (Lidarr alt / pasted custom URL) → `setArtwork`; a `songId` → writes that track's embedded image as the album folder's `cover.jpg` (`writeFolderCover`) and **deletes the canonical override** (`deleteArtwork` + `purgeDiskArtCache`) so file art is served — this is also the **revert-to-original** path. The picker shows the current cover instantly (synthetic option) so it never blocks on a slow Lidarr lookup, and emits `coverChanged` so album-detail refetches + cache-busts **without closing** the modal. → See [docs/metadata-optimize.md](docs/metadata-optimize.md).
- **On-demand track analysis (BPM + genre)**: the `track-info-sheet` drawer's **Analysis section** computes/verifies per-track metadata on demand. **BPM** — `library_songs.bpm` (additive column) is read from tags at scan time (`music-metadata` `common.bpm`, preserved on rescan via `bpm = COALESCE(excluded.bpm, …)`); `POST /api/library/songs/:id/analyze` returns an existing tag value or runs `track-analysis.ts` `analyzeBpm()` (ffmpeg decodes a ~90 s mono PCM slice → `music-tempo`), then **persists the result both to the DB and back into the file tag** (`audio-tags.ts` now writes `TBPM`/Vorbis `BPM`) so it survives rescans. **Genre** — `GET …/genre-suggestion` (`verifyGenre()` → Lidarr artist genres, diacritic-insensitive match, degrades to `null` when Lidarr is unconfigured) proposes a genre; admin-only `POST …/genre` writes the tag + updates `library_songs.genre`. Everything ffmpeg-dependent is gated on `ffmpegAvailable()`. Types (`BpmAnalysisResult`, `GenreSuggestion`) live in `@nicotind/core`. **Bulk backfill** mirrors the per-track buttons across the whole library via two scripts (the `backfill-years` vs `optimize-metadata` offline/Lidarr split): `scripts/analyze-bpm.ts` is **offline** (ffmpeg + music-tempo, reuses `analyzeBpm`; bounded `--concurrency`) and fills every `bpm IS NULL` song; `scripts/backfill-genre.ts` needs a **live Lidarr** (reuses `verifyGenre`, looking up **once per artist** via the pure `planGenreBackfill`/`groupSongsByArtist` in `services/track-backfill.ts`). Both are **dry-run unless `--apply`**, write to `library_songs.{bpm,genre}` **and** the file tag, log to `<dataDir>/*.log`, and resume on re-run (`library_genres` facet counts refresh on the next full scan). → See [docs/library-scanner.md](docs/library-scanner.md).
- **Lyrics (on-demand, plugin-sourced, editable)**: lyrics are fetched on demand from a **lyrics source plugin** and persist + can be user-edited. This added a **new `metadata` plugin kind** with a **`lyrics` capability** (`@nicotind/core` `plugin/{manifest,capabilities,index}.ts`: `LyricsCapability.fetchLyrics(LyricsQuery)→LyricsResult|null`; `validatePluginManifest` allows `lyrics` for `metadata` and **scopes the `defaultEnabled:true` ban to `acquisition` only** so a benign keyless source can default-on) — the documented "add a kind" recipe; the registry kernel is unchanged. First source: the **LRCLIB plugin** (`services/plugins/lrclib/`, pure-JS `fetch`, no key, returns both plain + synced LRC; `/api/get` exact match → `/api/search` fallback; injectable `fetchFn`), registered + `seedEnabled('lrclib',…)` in `index.ts` so it activates on first boot. Storage is the **`library_lyrics` side-table** keyed on the path-derived `songId` (`lyrics-store.ts`; `plain_text`/`synced_text`/`source`/`customized`/`updated_at`); plain text is **also written back to the file tag** (`audio-tags.ts` now writes/reads **ID3 USLT** + **Vorbis `LYRICS`**, like BPM) so it survives a move/transcode (synced LRC stays DB-only — no clean embedded standard). Routes (`routes/library.ts`, registry threaded via `LibraryRoutesOptions.pluginRegistry`): `GET …/lyrics` (any user → `LyricsDto`|null), `POST …/lyrics/fetch` (any user — keyless/benign; **returns a cached row unmodified**; on a DB miss it first **recovers lyrics from the file tag** (`source:'file-tag'` — the orphaned-row-after-transcode path), else queries `getEnabledWithCapability('lyrics')`, persists the first hit + writes the tag; **a `customized` user edit is never clobbered unless `force:true`**; `503` when no lyrics source enabled), `PUT …/lyrics` (admin — sets `customized=1`, `source='user'`, **clears synced** since an edited body no longer matches its timing), `DELETE …/lyrics` (admin reset). Web: the `track-info-sheet` drawer's **Lyrics section** (`data-testid="lyrics-section"`, fetch/`edit`("Add" when empty)/save/reset) and the **now-playing full-screen lyrics panel** (`now-playing-lyrics-toggle` swaps the queue for a panel that **highlights the active line** via the pure `lib/lrc-parser.ts` `parseLrc`/`findActiveLine` driven by `player.currentTime()`, falling back to scrollable plain text). `LyricsDto` lives in `@nicotind/core` (+ re-export in `web/src/types/core.ts`).
- **Unified search**: `GET /api/search?q=` queries the local library first (`LibrarySearchProvider`), fires slskd network search in parallel. Local results display above a divider, network results below with download actions. Local song hits carry `artistId` (selected from `library_songs.artist_id`, threaded through `SearchProviderResult.songs` → `toTrack`) so a track played from search resolves the player's now-playing **"Go to artist"** link to the real artist page instead of falling back to `/library`. **Network (slskd) hits have no `artistId`** (the artist isn't local yet), so the now-playing + downloads artist links resolve **by name** via `GET /api/library/artists/by-name?name=` (diacritic-insensitive lookup against `library_artists`; the pure `resolveArtistTarget()` in `lib/route-utils.ts`): known id → artist page, else name resolves to a local artist → their page, else `/library` (downloads falls back to a name search instead). Network (slskd) song results no longer live in a standalone "Songs" lane: `lib/song-results.ts` `groupBySong()` (dedupe + best-copy pick: FLAC > other lossless > highest-bitrate lossy, then peer availability) feeds `songResultToCandidate` and is **blended into the one source-agnostic Results list** (`data-testid="results"`, rows `acquire-result` + `source-chip`) alongside archive.org/Spotify candidates (from `GET /api/sources/search`), each with a single Get; the raw folder-tree browser stays under the demoted "Advanced" disclosure for whole-album peer grabs. The source-status line (`data-testid="source-status"`) is neutral ("Sources: …"), not "Soulseek network available". → See [docs/source-agnostic-acquisition.md](docs/source-agnostic-acquisition.md) and [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §F1.
- **Guided acquire UX (raw network demoted)**: the catalog cards (album/EP/single, all routing to the same `huntCatalogAlbum` → `AlbumHuntModalComponent`) are the primary path; the raw network lane — peer bitrate/speed/queue stats, the Songs↔Folders toggle, and the full folder-tree `FolderBrowserComponent` — lives **only inside a collapsed "Advanced: browse Soulseek peers & folders" disclosure** (`data-testid="advanced-toggle"`), auto-opened only when there's no catalog hit so it stays the escape hatch, never the default (the acquirable hits surface in the blended Results list above it — see the source-agnostic bullet). The album-hunt modal **leads with only the auto-selected best match** (`data-testid="hunt-best-match"`: "✓ Best match ready" + format/size/match%) and one Download action; the **full ranked peer list is opt-in** behind a collapsed **"Choose a different source (N found)"** disclosure (`data-testid="hunt-sources"`, holding the `hunt-candidate` rows + selection) so the common "just get it" case is a single tap, and its filters (FLAC/live/skew/min-match) are tucked behind a separate **"Explore more options"** `<details>` (`data-testid="hunt-options"`). The URL box is framed as **"Get from a link"** (auto-detects YouTube/SoundCloud/Bandcamp/Spotify/archive.org via `AcquireService.submit` — no backend choice exposed).
- **Inline download lifecycle**: Search result cards show idle → blue progress wash + % → green "▶ Open in Library". Driven by `TransferService`; a `libraryDirty` signal tracks completed downloads (cleared on library page load, ignoring initial boot completions to prevent double-loading and disruptive auto-refreshes of active views). `TransferService` uses adaptive polling: 3 s when any transfer or acquire job is active, 30 s when idle. `kickPoll()` resets to fast mode immediately when a download is initiated.
- **Multi-user**: Shared music library (all users see all downloads). Per-user settings in bun:sqlite (`packages/api/src/db.ts`). First registered user becomes admin.
- **Remote playback (cast to another device, Spotify-Connect-style)**: a per-user `PlaybackStateManager` (`services/playback-state.ts`, one per user via `playback-registry.ts`) holds the connected-device map, the single active-output device, and the shared playback state, broadcasting `STATE_SYNC`/`DEVICES_SYNC`/`COMMAND` over the authed `GET /api/ws/playback` WebSocket (`services/websocket.ts`; JWT in the query param). Each browser tab is a **device** (stable `nicotind_device_id` in localStorage); it opts in via **Settings → Remote Playback** (`data-testid="remote-toggle"`), which both registers it as an available output and supplies the user-activation Chrome needs to autoplay an incoming cast. The web mirrors server state as signals in `RemotePlaybackService` (+ `PlaybackWsService`: reconnect/backoff, 30 s heartbeat, auto-disable after 5 persistent failures); the **device switcher** (`DeviceSwitcherComponent`, the cast button — `data-testid="device-switcher-toggle"`/`-panel`/`device-option`/`device-now-playing`) is **mounted twice** (player bar + now-playing sheet) sharing the service-level `switcherOpen` signal, so both popovers open together (scope e2e selectors with `.first()`). Selecting a device sends `SET_ACTIVE_DEVICE` + `SET_TRACK` and closes the popover; thereafter the controller's transport buttons relay `COMMAND` `PLAY`/`PAUSE`/`SEEK`/`NEXT`/`PREV` (only the active, opted-in device executes — echo-protected via `lastRemoteTrackId`). A 90 s stale timeout (3 missed heartbeats) unregisters a device and clears the active output. Covered by the **two-context** `remote-playback` playground flow (controller + target speaker, distinct device ids seeded per context) — see docs/testing-routines.md.
- **Service modes**: `embedded` (NicotinD spawns slskd as child process) or `external` (connects to a pre-existing slskd). Library/streaming stack is in-process — no music-server subprocess.
- **Auth flow**: NicotinD issues its own JWTs; holds auto-generated API key for slskd. Web player streams via `/api/stream/:id?token=` (JWT in query param). **Sliding sessions**: the JWT TTL defaults to **30 days** (`jwt.expiresIn`, `config/default.yml`) and the web silently renews it on every app boot — `provideAppInitializer` in `app.config.ts` fires `ApiService.refreshToken()` → `POST /api/auth/refresh` (`routes/auth.ts`, guarded by the existing `authMiddleware`, re-signs via `signJwt`) and swaps the token in via `AuthService.setToken()` (token-only, leaves username/role). So an active user effectively never hits the expiry wall and stops re-logging in daily; an actually-expired token 401s on refresh and the interceptor falls back to `/login`. **Share tokens are non-refreshable** (`/refresh` 403s on `share === true`) — they stay short-lived and read-only. The login page (`pages/login/`) is a fixed-palette glassmorphic card (indigo/fuchsia/teal aurora, theme-independent).
- **Release-type model (singles & EPs, Spotify-style)**: every album carries a `classification` (`album`/`ep`/`single`/`compilation`/`unknown`). The `LibraryCurator` classifies **metadata-first** — Lidarr/MusicBrainz `albumType` from the `library_release_meta` side table (`release-meta-store.ts`, keyed on the scanner's `albumId`, survives rescans) wins; otherwise a track-count heuristic (1 → single, 2–6 → ep, 7+ → album). Album-less tracks are **un-bucketed at scan time**: `isLooseSinglesBucket` makes each loose track its own single release named after the title (no more shared hidden `<Artist>/Singles/` bucket; the organizer no longer force-writes `album="Singles"`). The Albums grid stays album-only via a single centralized filter (`GRID_CLASSIFICATION_SQL` in `routes/library.ts`); singles & EPs surface on the **artist page** (`GET /artists/:id` → `{ albums, singlesAndEps }`) and a dedicated **`GET /library/singles`** list. Ingest-time `SingleEnrichmentService` does a best-effort Lidarr lookup on URL-acquired singles (release type + canonical artwork via the existing `artwork-store`), degrading gracefully when Lidarr is unconfigured. → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Native playlists (per-user)**: `playlists`/`playlist_songs` tables + `PlaylistService` (`services/playlist.service.ts`) behind `/api/playlists` (`routes/playlists.ts`), every handler scoped to `c.var.user.sub` so playlists are private. The **archive.org** plugin (`services/plugins/archive/`) is a third `resolve` backend but **pure JS** (no binary): it reads an item's `/metadata/<id>` file list, picks one audio format (MP3-preferred, FLAC fallback; `acquire.archive.preferredFormats`), and streams those files into staging. It also backs a **search lane** (`ArchiveSearchService` → `advancedsearch.php`, `GET /api/archive/search?q=` or `?artist=&album=`) gated specifically on the `archive` plugin (`plugins.isEnabled('archive')` → `503`, independent of slskd). Queries are **field-targeted + phrase-quoted** (`title:`/`creator:`) not bare full-text, **exclude the non-music audio collections** (`-collection:(librivoxaudio OR audiobooksandpoetry OR audio_bookspoetry OR oldtimeradio OR radioprograms OR radio OR podcasts OR audio_religion OR audio_news OR audio_tech OR gratefuldead OR etree)` — the broader `audiobooksandpoetry` umbrella + live-tape archives (`gratefuldead`/`etree`) flood "best of" queries, so a "Shaggy" search doesn't return a LibriVox audiobook), and are **`sort[]=downloads desc`** so popular real music outranks obscure mashups; results are then **deduped** by the pure `archiveDedupeKey()` (diacritic-folded, bracket/format/year noise stripped → sorted token set, so `Porfiado · El Cuarteto De Nos` and `El Cuarteto de Nos - Porfiado (2012) [FLAC]` collapse to one, popularity-sorted so the best copy survives). Each deduped item (capped at `MAX_ENRICH`) is then **enriched with a track count + `kind` (album/single)** via a bounded parallel `/metadata/<id>` lookup — the pure `countArchiveTracks()` takes the largest single-format audio group (so a FLAC+MP3 dual-encode counts once); items the lookup **proves have no audio files are dropped** (extra junk removal), a failed lookup leaves the count `null` (graceful). The web renders `creator · year · N tracks · album/single` via the pure `lib/archive-display.ts` `archiveSubtitle()`, omitting any absent piece (no literal "Unknown") so the user sees whether an item is a single or a multi-track album before acquiring. The service **retries once and throws `ServiceUnavailableError`** on an unreachable/non-OK upstream (route → `503`) so a transient archive.org failure shows as "unavailable" rather than a misleading empty result. → See [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §B. The web surfaces it in **two places** (both download by handing the item's `detailsUrl` to `/api/acquire`): the **album-hunt modal** (an "Also on archive.org" section run in parallel with the Soulseek hunt) and the **main unified search** ("From archive.org" section); both gate on `PluginService.hasArchive`. **Default-off compliance**: fresh installs enable nothing (`seedLegacyAcquisitionPlugins` migrates _existing_ installs once, guarded by an `app_settings` marker; archive is **not** seeded so it stays off for every install). **Web**: `PluginService` (signals, capability computeds `hasSearch`/`hasResolve`/`hasDownload` + `hasArchive`) + the admin **Settings → Plugins** page (`pages/plugins/`, route `/settings/plugins`); the search page gates the URL acquire box on `hasResolve` and the watchlist star on `hasDownload`. _Phases A–D landed; connectivity (E) is scaffolded (kind handled generically) but no connectivity plugin is shipped._ → See [docs/plugins.md](docs/plugins.md).

## Mobile app (Capacitor Android + iOS)

`packages/mobile` is a thin **Capacitor** shell that wraps the **same** `@nicotind/web` Angular build for Android — no second UI codebase (the Immich mentor ships native Flutter; we wrap to avoid double work for one maintainer, keeping only Immich's **server-URL entry screen** idea). The enabler is a **runtime-configurable API base URL**: the web historically assumed same-origin `/api/*`; now `ServerConfigService` (`apiUrl()`/`wsUrl()`, base `''` on web = unchanged, defaults to the canonical self-hosted server on native — platform detected via Capacitor's injected global in `lib/platform.ts`, **no `@capacitor/core` dep in the web bundle**) + the `authInterceptor` rewrite cover all HttpClient calls; direct URLs (stream/cover/Media-Session art in `player.component.ts`, the playback WS, `preserve.service.ts`, the admin log `EventSource`, share view) call `apiUrl()`/`wsUrl()`, and **cover `<img>`s rewrite centrally in `CoverArtComponent` (`resolvedSrc`)**. A **server-picker** (`pages/server-config/`, route `/server`, `serverGuard`) validates against `GET /api/health`; it's **native-only** (`needsConfiguration()` is always false on web, so the web/e2e are unaffected) and the service worker is disabled on native. The WebView is cross-origin, so the API mounts `nativeAppCors()` (`packages/api/src/middleware/cors.ts`) allowing the localhost Capacitor origins + `Authorization`/`Range` and **exposing the 206 range-streaming headers**. Android version is driven from `package.json` via `androidVersion()` (`packages/mobile/src/version.ts`) + `scripts/android-env.ts`; the CI `android` job (in `deploy.yml`, gated like `deploy`, **JDK 17**) builds the web, `cap sync`s, signs `assembleRelease` (keystore from secrets) and attaches the APK to the GitHub Release. **Background audio + lock-screen/notification controls** (the key native value-add) are handled by **`@jofr/capacitor-media-session`** — the Android WebView ignores the Web Media Session API and suspends backgrounded audio, so the plugin provides a native `MediaSession` + media-playback **foreground service**; it's wrapped by a lazily-imported `MediaControlsService` and fed by the pure `buildMediaMetadata` (`lib/media-metadata.ts`), with `player.component.ts` driving metadata/playback-state/handlers/position. The plugin officially targets **Capacitor 6**, so `packages/mobile` is pinned there (revisit on Cap 7+ support). The **native launcher/app icon AND the launch/splash screen are the NicotinD brand mark** (same SVG as the PWA manifest icon/favicon, not Capacitor's default bolt): one source in `src/native-icons.ts` (pure, unit-tested builders — `fullIconSvg`/`backgroundSvg`/`foregroundSvg` for the icon layers, `splashSvg` for the centred dark splash) → `scripts/generate-native-icons.ts` rasterizes the committed `assets/*.png` sources (icons 1024², splashes 2732²) via `sharp` → **`@capacitor/assets`** generates the Android mipmaps + splash drawables (committed) + iOS AppIcon + splash (generated in CI, since `ios/` is ephemeral). → See [docs/mobile-app.md](docs/mobile-app.md).

- **iOS (same Capacitor shell)**: the whole native-enablement layer above is **platform-neutral** and reused unchanged on iOS — `isNativePlatform()`, `ServerConfigService`/server-picker, `MediaControlsService` (the `@jofr` plugin is a Web Media Session wrapper on iOS), and `nativeAppCors()` (which **already allow-lists `capacitor://localhost`**, the iOS WKWebView origin — no API change). iOS-specific bits: `iosVersion()` (`packages/mobile/src/version.ts`, shares the monotonic scheme → `CFBundleShortVersionString`/`CFBundleVersion`) + `scripts/ios-env.ts`; an ephemerally-generated `ios/` project (Xcode/CocoaPods need a Mac, so it's **not committed** — CI runs `cap add ios` each run and patches `Info.plist` via `scripts/ios-plist.ts`/`src/ios-plist.ts` for `UIBackgroundModes: [audio]`); the macOS CI `ios` job (`deploy.yml`, gated like `deploy`) builds an **unsigned** IPA (`CODE_SIGNING_ALLOWED=NO`, no Apple account yet) and attaches it to the Release (install via AltStore/Sideloadly). **Background audio on iOS works**, but the `@jofr` plugin ships **no iOS native code** (its `dist/` only wraps WKWebView's Web Media Session API), and WKWebView wires play/pause to the `<audio>` element but does **not** surface JS-set metadata/artwork/position for cross-origin web audio — so the iOS lock-screen card showed controls but **no title/artist/artwork/time**. Fix: a minimal iOS-only Capacitor plugin **`@nicotind/capacitor-now-playing`** (`packages/capacitor-now-playing/`, Swift `NowPlayingPlugin` → `MPNowPlayingInfoCenter`) owns the *displayed info* (title/artist/album/artwork/duration/elapsed + `playbackState`); `MediaControlsService` routes `setMetadata`/`setPlaybackState`/`setPositionState` to it **only when `isIosNative()`** (via `getCapacitorPlugin('NicotindNowPlaying')` off the injected `Capacitor` global — no new web import; pure mapping `toNativeMetadata`/`pickArtworkUrl` in `lib/now-playing.ts`), falling back to `@jofr` if absent. **The plugin also owns transport — this is the fix for the card that never actually worked**: the original "just set `nowPlayingInfo`, let WKWebView keep transport, register no `MPRemoteCommandCenter`" design couldn't work, because iOS only *displays* `nowPlayingInfo` for the app that owns the now-playing session, which needs an **active `AVAudioSession` + ≥1 registered `MPRemoteCommandCenter` target** — both held by WKWebView, so the system read WebKit's empty session. So the plugin now activates `AVAudioSession(.playback)` (`ensureSession`, lazy on first play) and registers the lock-screen commands (`registerCommands`), forwarding each to JS via a single `remoteCommand` event. **Consequently iOS bypasses `@jofr` `setActionHandler`** — `MediaControlsService.setActionHandler` branches on `isIosNative()` and dispatches the native `remoteCommand` event to per-action handlers (wiring both would double-fire every press). The plugin is a workspace dep of `@nicotind/mobile` so `cap sync ios` discovers + pods it (no `deploy.yml` change); the Swift is build-gated by the macOS `ios` job. **No macOS CI test job** (cost) — the Swift is verified on-device via a `getDiagnostics()` plugin method surfaced in the **Settings → "Now Playing (iOS)"** panel (native-iOS-only, `data-testid="now-playing-diagnostics"`: plugin registered? session/category? commands registered? now-playing keys? last artwork status?), with `artworkError` events surfacing ATS/network artwork failures. Residual risk (device-only): activating our `AVAudioSession` could disturb the working background playback (mitigated: `.playback`, lazy activate, `setActive(false)` only on `clear()`). Two adjacent fixes shipped with this: (1) **safe-area header** — the sticky header sat *under* the iOS notch, fixed by `viewport-fit=cover` (`index.html`) + folding `env(safe-area-inset-top)` into the header's top padding (no-op on web); (2) **`MediaControlsService` `@jofr` proxy fix** — `session()` used to resolve its promise to the bare Capacitor `MediaSession` proxy, and Promise resolution probing `.then` on the proxy threw `MediaSession.then() is not implemented on web` (a real uncaught rejection in the production web console); it's now **boxed** (`{ session }`) so the proxy never lands on a resolution path. → See [docs/ios-app.md](docs/ios-app.md) and [docs/web-ui.md](docs/web-ui.md).

## End-to-end tests

`packages/e2e` is a Playwright suite that boots the real server (`bun run src/main.ts`) against a throwaway DB + committed silent-FLAC fixtures and drives the SPA in Chromium: auth, library, playback, **player controls (pause/next/seek/shuffle)**, and the compliance-critical plugin capability gating. No slskd/Lidarr needed — acquisition is default-off, so the test server runs `NICOTIND_MODE=external` with dead slskd/Lidarr URLs and a fresh `.tmp-data` dir per run (deterministic first-user admin). Selectors are `data-testid` attributes — **adding a `data-testid` is the standard for new e2e-targeted elements** (`app-password-field` forwards a `testId` input). Runs on port 8585 (override `E2E_PORT`); `E2E_BASE_URL` points it at a running instance (prod smoke) and skips the managed server. **CI is split across two workflows**: `ci.yml` ("CI & Release") runs the `ci` + `e2e` jobs on every push/PR (the `ci` job also **actionlint**s the workflow files themselves — a malformed `${{ }}` expression invalidates a whole workflow and fails it at startup with zero jobs, silently skipping deploy, so it's gated here), then a `release` job bumps the version and pushes a `vX.Y.Z` tag; that tag triggers `deploy.yml` ("Build & Deploy"), which SSH-deploys `git reset --hard <tag>` + `docker compose up`. The `android`/`ios` mobile-asset jobs in `deploy.yml` are **tag-only + `continue-on-error: true`** (best-effort Release artifacts — a flaky unsigned macOS build must not red-flag a successful server `deploy`). `deploy.yml` also accepts `workflow_dispatch` (manual redeploy of `origin/master`). A separate **gated playground harness** (`PLAYGROUND=1`, `tests/*.playground.ts`) drives real user flows against a live backend and **records observations** (timings, gaps, enhancement signals, cover-art 404s) into an aggregated findings report (`packages/e2e/playground-report/*.{md,json}`) via a custom reporter — the automated successor to the manual playground sessions. It degrades gracefully (never red on a dead backend) and stays out of CI; only its pure logic (`playground/{observe,report,net-monitor,console-monitor,journey}.ts`) is unit-tested in the `ci` job. **Each flow also captures runtime health and friction**: the fixture wires `page.on('console')`/`pageerror`/`requestfailed` → `console-monitor.ts` (a `console.error` during a normal flow is a defect signal even when every HTTP call was 200; a small ignore list drops the browser's generic "Failed to load resource" echo the response classifier already records), and `obs.journey()` (step/fallback/deadEnd, auto-flushed) + `obs.outcome(...)` feed a per-flow **Outcome matrix** + **Health summary** in the report. Gap-fill flows cover **downloads**, **playlists** (self-cleaning), **sharing** (anonymous read-only, self-cleaning) and **admin/plugins** (read-only; toggle gated by `PLAYGROUND_PLUGIN_TOGGLE`). The **real round-trip** (`tests/real-roundtrip.real.ts` via `playwright.real.config.ts`, `bun run … playground:real`) is the only mutating tier — it **acquires → verifies in-library → verifies playable → DELETEs** against prod with always-run cleanup, behind a two-key guard (`E2E_BASE_URL` **and** `PLAYGROUND_REAL=1`); never in CI. A parallel family of **screenshot flows** (`tests/*.screens.ts`, also out of CI) captures per-state mobile shots for UX review under `screenshots/mobile/<flow>/` (pure `playground/shot.ts` helper, CI-unit-tested): `playwright.live-screens.config.ts` (`bun run --filter @nicotind/e2e screens:live`) drives the live-backend **player-analysis** + **downloads-acquire** mobile journeys and — because they use the `obs` fixture — emits screenshots *and* the findings report in one run (mutating sub-steps gated behind `PLAYGROUND_ANALYZE` / `PLAYGROUND_ACQUIRE_URL`). **The flow catalogue (every main flow × which tier covers it) and the recurring routines (weekly live pass, pre-release real round-trip, screenshot pass, monthly log rotation) live in [docs/testing-routines.md](docs/testing-routines.md).** → See [docs/e2e.md](docs/e2e.md) "Playground harness" / "Real round-trip" / "Screenshot flows" and [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §E2/§F.

**Real-use feedback log**: [docs/feedback-log-2026-06.md](docs/feedback-log-2026-06.md) is a rolling, dated log of friction noticed while actually *using* the app (distinct from the automated playground findings) — one entry per observation with Severity/Status, so recurring pain clusters surface as a prioritization signal ("is this tool useful?"). Rotate monthly.

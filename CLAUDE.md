# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Quality Gates

Every task on this project must satisfy all three gates before being considered done:

1. **Every change must be tested.** New features get new tests. Bug fixes get regression tests. Refactors must not reduce coverage. If a change can't reasonably be unit-tested, add an integration or e2e test instead — untested code is not shippable.

2. **Every test must run in CI.** Adding a test locally is not enough. Verify the relevant GitHub Actions workflow actually executes the new test on push. If a new test file or package is added, confirm it's picked up by `.github/workflows/`. Don't close out a task until CI covers the new test.

3. **Documentation must be updated in the same change as the code.** This is not optional and not a follow-up task: **every time you add or modify behavior, update the docs in the same commit/PR.** Significant decisions — new patterns, new services, why an approach was chosen over alternatives, trade-offs accepted — must be captured. If a change makes an existing doc statement wrong, fix that statement; stale docs are treated as a bug. Where docs live:
   - **Inline in `CLAUDE.md`** (the default) — add or update a bullet in the relevant section (Key Design Patterns / Web UI). This is the established pattern; most subfeatures are one dense bullet here.
   - **A dedicated `docs/<feature>.md` file** when a feature is large enough that an inline bullet can't do it justice. When you do this, **add a one-line pointer to it from `CLAUDE.md`** so the main file remains the index — a reader should never have to discover a `docs/` file by accident.
   - **A concise `// why` comment in code** for local rationale that belongs next to the implementation.

   A change is not "done" (gate-complete) until its documentation reflects reality.

## What is NicotinD?

NicotinD is a unified music acquisition + streaming platform that orchestrates **slskd** (Soulseek P2P client) behind a single API, web UI, and CLI, and **natively scans/streams** the music library itself (Navidrome was removed — see Architecture). Downloads from Soulseek land in a shared folder; the DownloadWatcher organizes and incrementally scans completed transfers into the canonical SQLite library that the API streams from. URL-based acquisition (yt-dlp / spotdl) feeds the same pipeline.

## Commit Conventions

This repo uses [Conventional Commits](https://www.conventionalcommits.org/). Every commit message must follow this format:

```
<type>(<optional scope>): <description>
```

**Types that bump the version** (appear in CHANGELOG):
| Type | When to use | Version bump |
|------|-------------|--------------|
| `feat` | New user-facing feature | minor |
| `fix` | Bug fix | patch |
| `perf` | Performance improvement | patch |

**Types that don't bump** (hidden from CHANGELOG):
| Type | When to use |
|------|-------------|
| `chore` | Deps, tooling, config, CI tweaks |
| `refactor` | Code restructuring (no behavior change) |
| `style` | Formatting, whitespace |
| `docs` | Documentation only |
| `test` | Adding/updating tests |
| `ci` | CI pipeline changes |
| `build` | Build system changes |

**Breaking changes**: Add `BREAKING CHANGE:` in the commit body or `!` after the type (e.g. `feat!: remove legacy auth`) to trigger a major bump.

**Enforcement**: A `commit-msg` hook via husky + commitlint rejects non-conforming messages.

**Releasing**: When ready to release, run `bun run release`. It reads the commit history since the last tag, determines the version bump, updates `package.json`, generates/updates `CHANGELOG.md`, commits, and creates a git tag.

## Architecture

```
NicotinD (Hono API :8484)  — native library scanner + streaming, all in-process
└── slskd (Soulseek client :5030)  ──── shared /data/music folder
        DownloadWatcher → LibraryOrganizer → LibraryScanner (tags → SQLite)
```

> **Navidrome was removed.** NicotinD is now fully native: it scans the music
> dir itself (`LibraryScanner`, `music-metadata`), serves audio bytes from disk
> with HTTP range support (optional ffmpeg transcoding), and resolves cover art
> from folder/embedded images. The canonical `library_*` SQLite tables are the
> single source of truth. The **`/rest/*` Subsonic proxy and the playlist feature
> were dropped** in the same migration (playlists to be re-added natively later).

**Bun monorepo** with workspace packages:

| Package                     | Purpose                                                                                                             |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `@nicotind/core`            | Shared types (Zod schemas), logger (pino), crypto utils, error classes                                              |
| `@nicotind/slskd-client`    | Typed HTTP client wrapping slskd's REST API (`/api/v0/*`)                                                           |
| `@nicotind/service-manager` | Strategy pattern for managing sub-service lifecycle (child_process or Docker)                                       |
| `@nicotind/api`             | Hono API server — routes, JWT auth, unified search, download watcher, native library scanner + streaming, SQLite DB |
| `@nicotind/web`             | Angular v22 web UI (standalone components, signals, Tailwind)                                                       |
| `@nicotind/cli`             | Commander.js CLI (Phase 3)                                                                                          |

**Entry point**: `src/main.ts` — loads config, starts services, wires clients into the API server.

## Key Design Patterns

- **Native library scanner**: `LibraryScanner` (`packages/api/src/services/library-scanner.ts`) walks the music dir, reads tags via `music-metadata`, and writes to `library_*` SQLite tables with deterministic SHA1 IDs. Includes clean tracklist selection (one best file per track) via `selectAlbumTracks`, Lidarr-canonical matching, and edition-collapsing album IDs. `resolveTags` (threaded through `buildLibrary`/`selectLibraryTracks` as a pure `overrides` map, loaded from `library_metadata_overrides`) substitutes user-confirmed artist/album/year corrections **before** minting the artistId/albumId, so a fix re-buckets the album and survives full rescans (see the user-driven metadata fix bullet). → See [docs/library-scanner.md](docs/library-scanner.md).
- **Native streaming + cover art**: `streamingRoutes` serves `GET /api/stream/:id` from disk with HTTP Range/206 and optional ffmpeg transcoding; `GET /api/cover/:id` resolves canonical → folder → embedded art. 404 responses carry `Cache-Control: public, max-age=300` and a module-level `noArtCache` (10 min TTL) short-circuits the slow `extractCover()` IO on repeat misses. **Transcoded streams are seekable**: on-the-fly transcoding used to be a sequential ffmpeg *pipe* (status 200, no `Content-Length`/`Accept-Ranges`), so far seeks did nothing (the iOS/Firefox "seek to the end does nothing" bug — acute because the Opus library standardization means iOS *needs* transcoding to play at all). `transcode-cache.ts` (`getTranscodedFile`) now transcodes the whole file **once** to a disk cache (`<dataDir>/transcode-cache`, keyed on source path+**mtime**+format+bitrate so a re-encode invalidates it; concurrent plays de-duped via an in-flight map; oldest-first eviction over a 2 GiB soft budget) and **both** the transcode path and the passthrough path serve through the shared `serveFileWithRange()` helper — so every stream advertises ranges and seeks correctly. `transcode.ts` `transcodeToFile()` writes atomically (temp + rename). First play of a track warms the cache (one transcode); replays are instant and seekable. → See [docs/library-scanner.md](docs/library-scanner.md).
- **Canonical artwork**: `library_artwork` table stores canonical URLs keyed on deterministic IDs (survives rescans); populated by `hunt-download` and `scripts/backfill-artwork.ts`. → See [docs/library-scanner.md](docs/library-scanner.md).
- **Metadata optimization (fix wrong/poor covers + year)**: `metadata-optimize.ts` re-fetches better cover/year/release-type from Lidarr and **overwrites** what's stored — distinct from `backfillArtwork`, which only fills *missing* art. `optimizeAlbum()` matches the global `album.lookup("<artist> <title>")` (normalized title+artist, reusing `looksLikeNonAlbum`/`normalizeName` from `artwork-backfill.ts`), then `setArtwork` (purges the stale cover cache), updates `library_albums.year` (placeholder `0001` dropped), and `setReleaseType`. This **automatic** matcher is all-or-nothing and intentionally conservative: it's the cheap **bulk** backfill (`optimizeAllAlbums`, default-scoped to albums missing artwork/year; `POST /api/admin/metadata-optimize?all=1&dryRun=1` + admin Settings button + `scripts/optimize-metadata.ts`). It **skips placeholder-artist albums** (`isPlaceholderArtist` in `artwork-backfill.ts` — `<Desconocido>`/`Unknown`/`Various Artists`/bracketed): the artist guard could never pass and the lookup would be poisoned, so those are deferred to the user-driven fix below. → See [docs/metadata-optimize.md](docs/metadata-optimize.md).
- **User-driven metadata fix (correct a wrong artist/album; confirm low-confidence)**: the all-or-nothing matcher fails when the *stored artist itself is wrong* — a rip tagged `<Desconocido>` searches `"<Desconocido> Selva"`, a **poisoned query** a known band never matches, leaving the wrong cover. `metadata-fix.ts` is the interactive answer: `searchCandidates()` runs an **editable** Lidarr query (default `"<artist> <album>"`, but **album-title-only when the stored artist is a placeholder** per `isPlaceholderArtist` — so `<Desconocido>` searches `"Selva"` and actually returns La Portuaria; the modal shows an amber hint prompting the user to type the real artist, and the box stays user-overridable) and `rankCandidates()` (pure, diacritic-folded `NFD` token overlap) returns the best-first top 8 **including low-confidence hits the user confirms**; `applyMetadataFix()` applies a confirmed candidate **or free-text** (`{artist?,album?,year?,coverUrl?,releaseType?}`). The correction persists in **`library_metadata_overrides`** (keyed on the scanner's **raw** `albumId`; `corrected_album_id` reverse-lookup via `findByCorrectedId` avoids orphaning a row on re-correction) which `resolveTags` consults at scan time, and the canonical tables are mutated **immediately** to match: songs are UPDATEd in place (`songId` is path-derived → `starred`/`hidden`/`playlist_songs` survive), the album row + `library_artwork`/`library_release_meta` are re-pointed to the corrected id, the corrected artist is upserted and the orphaned old artist pruned via the shared `pruneOrphanArtist` (`services/library-aggregates.ts`, also reused by album-delete). Admin-only: `GET /api/library/albums/:id/metadata-candidates?q=` (needs Lidarr) + `POST /api/library/albums/:id/metadata` (free-text works offline) → album-detail **"Fix metadata"** button opens `MetadataFixModalComponent` (`data-testid="optimize-metadata"`); on apply the page re-fetches the corrected album by its (possibly new) id and cache-busts the cover. → See [docs/metadata-optimize.md](docs/metadata-optimize.md).
- **On-demand track analysis (BPM + genre)**: the `track-info-sheet` drawer's **Analysis section** computes/verifies per-track metadata on demand. **BPM** — `library_songs.bpm` (additive column) is read from tags at scan time (`music-metadata` `common.bpm`, preserved on rescan via `bpm = COALESCE(excluded.bpm, …)`); `POST /api/library/songs/:id/analyze` returns an existing tag value or runs `track-analysis.ts` `analyzeBpm()` (ffmpeg decodes a ~90 s mono PCM slice → `music-tempo`), then **persists the result both to the DB and back into the file tag** (`audio-tags.ts` now writes `TBPM`/Vorbis `BPM`) so it survives rescans. **Genre** — `GET …/genre-suggestion` (`verifyGenre()` → Lidarr artist genres, diacritic-insensitive match, degrades to `null` when Lidarr is unconfigured) proposes a genre; admin-only `POST …/genre` writes the tag + updates `library_songs.genre`. Everything ffmpeg-dependent is gated on `ffmpegAvailable()`. Types (`BpmAnalysisResult`, `GenreSuggestion`) live in `@nicotind/core`. → See [docs/library-scanner.md](docs/library-scanner.md).
- **Unified search**: `GET /api/search?q=` queries the local library first (`LibrarySearchProvider`), fires slskd network search in parallel. Local results display above a divider, network results below with download actions. Local song hits carry `artistId` (selected from `library_songs.artist_id`, threaded through `SearchProviderResult.songs` → `toTrack`) so a track played from search resolves the player's now-playing **"Go to artist"** link to the real artist page instead of falling back to `/library`. **Network (slskd) hits have no `artistId`** (the artist isn't local yet), so the now-playing + downloads artist links resolve **by name** via `GET /api/library/artists/by-name?name=` (diacritic-insensitive lookup against `library_artists`; the pure `resolveArtistTarget()` in `lib/route-utils.ts`): known id → artist page, else name resolves to a local artist → their page, else `/library` (downloads falls back to a name search instead). Network results default to a **song-first "Songs" lane** (toggle: Songs ↔ Folders) — pure `lib/song-results.ts` `groupBySong()` dedupes the flat slskd file list by normalized `(artist, title)`, auto-picks the best copy (FLAC > other lossless > highest-bitrate lossy, then peer availability) and one-click downloads it via the existing `enqueueDownload`; the folder view stays for whole-album grabs. This is the answer to "find me one song" (the curated catalog/hunt tools are album/EP-only). → See [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §F1.
- **Guided acquire UX (raw network demoted)**: the catalog cards (album/EP/single, all routing to the same `huntCatalogAlbum` → `AlbumHuntModalComponent`) are the primary path; the raw network lane — peer bitrate/speed/queue stats, the Songs↔Folders toggle, and the full folder-tree `FolderBrowserComponent` — lives **only inside a collapsed "Advanced: search the network directly" disclosure** (`data-testid="advanced-toggle"`), auto-opened only when there's no catalog hit so it stays the escape hatch, never the default. The album-hunt modal leads with the ranked result + one Download action; its filters (FLAC/live/skew/min-match) are tucked behind an **"Explore more options"** `<details>` (`data-testid="hunt-options"`). The URL box is framed as **"Get from a link"** (auto-detects YouTube/SoundCloud/Bandcamp/Spotify/archive.org via `AcquireService.submit` — no backend choice exposed).
- **Inline download lifecycle**: Search result cards show idle → blue progress wash + % → green "▶ Open in Library". Driven by `TransferService`; a `libraryDirty` signal tracks completed downloads (cleared on library page load, ignoring initial boot completions to prevent double-loading and disruptive auto-refreshes of active views). `TransferService` uses adaptive polling: 3 s when any transfer or acquire job is active, 30 s when idle. `kickPoll()` resets to fast mode immediately when a download is initiated.
- **Multi-user**: Shared music library (all users see all downloads). Per-user settings in bun:sqlite (`packages/api/src/db.ts`). First registered user becomes admin.
- **Service modes**: `embedded` (NicotinD spawns slskd as child process) or `external` (connects to a pre-existing slskd). Library/streaming stack is in-process — no music-server subprocess.
- **Auth flow**: NicotinD issues its own JWTs; holds auto-generated API key for slskd. Web player streams via `/api/stream/:id?token=` (JWT in query param). **Sliding sessions**: the JWT TTL defaults to **30 days** (`jwt.expiresIn`, `config/default.yml`) and the web silently renews it on every app boot — `provideAppInitializer` in `app.config.ts` fires `ApiService.refreshToken()` → `POST /api/auth/refresh` (`routes/auth.ts`, guarded by the existing `authMiddleware`, re-signs via `signJwt`) and swaps the token in via `AuthService.setToken()` (token-only, leaves username/role). So an active user effectively never hits the expiry wall and stops re-logging in daily; an actually-expired token 401s on refresh and the interceptor falls back to `/login`. **Share tokens are non-refreshable** (`/refresh` 403s on `share === true`) — they stay short-lived and read-only. The login page (`pages/login/`) is a fixed-palette glassmorphic card (indigo/fuchsia/teal aurora, theme-independent).
- **Release-type model (singles & EPs, Spotify-style)**: every album carries a `classification` (`album`/`ep`/`single`/`compilation`/`unknown`). The `LibraryCurator` classifies **metadata-first** — Lidarr/MusicBrainz `albumType` from the `library_release_meta` side table (`release-meta-store.ts`, keyed on the scanner's `albumId`, survives rescans) wins; otherwise a track-count heuristic (1 → single, 2–6 → ep, 7+ → album). Album-less tracks are **un-bucketed at scan time**: `isLooseSinglesBucket` makes each loose track its own single release named after the title (no more shared hidden `<Artist>/Singles/` bucket; the organizer no longer force-writes `album="Singles"`). The Albums grid stays album-only via a single centralized filter (`GRID_CLASSIFICATION_SQL` in `routes/library.ts`); singles & EPs surface on the **artist page** (`GET /artists/:id` → `{ albums, singlesAndEps }`) and a dedicated **`GET /library/singles`** list. Ingest-time `SingleEnrichmentService` does a best-effort Lidarr lookup on URL-acquired singles (release type + canonical artwork via the existing `artwork-store`), degrading gracefully when Lidarr is unconfigured. → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Native playlists (per-user)**: `playlists`/`playlist_songs` tables + `PlaylistService` (`services/playlist.service.ts`) behind `/api/playlists` (`routes/playlists.ts`), every handler scoped to `c.var.user.sub` so playlists are private. Songs are referenced by the scanner's stable `songId`; reads JOIN `library_songs` and **silently drop songs whose file moved** (id changed) so a playlist degrades rather than showing dead entries. → See [docs/web-ui.md](docs/web-ui.md).
- **Catalog (metadata-driven) search**: `CatalogService` (`/api/catalog`) looks up Lidarr/MusicBrainz and returns structured artist/album cards; resolves to the album-hunt flow on selection. Raw slskd search is an always-visible fallback. `search()` **scopes the album cards to the matched artist** when that artist's own releases appear in the lookup (the global `album.lookup` otherwise ranks mashups/tributes/compilations that merely contain the query above the real discography), falling back to all albums for a pure album-title search; artist pills are **deduped by normalized name**. When the query **exact-matches an artist but the lookup carries none of *their* albums** (e.g. "Zara Larsson" → only mashups/tributes), `search()` returns **0 cards** + `discographyUnavailable`/`scopedArtist` (not the junk that all 404s on resolve); the web suppresses the grid, **auto-opens the network lane** (`shouldOpenDirectSearch`), and offers a **"Load <artist>'s discography"** button → `loadDiscography()` (`POST /api/catalog/discography`) **adds the artist to Lidarr on demand** and lists their real, hunt-able albums (the §A6 deep fix — user-initiated, never automatic). Own-artist cards are ranked Album > EP > Single, newest-first. `resolveAlbum` matches the card's release-group id against the artist's Lidarr discography and **falls back to a diacritic-insensitive title match** (cards come from the *global* `album.lookup`, whose ids aren't all present in `listByArtist` — esp. compilations/"best of"), throwing a typed `404` (`ALBUM_NOT_IN_LIDARR`) rather than a `500` when truly absent. The resolve route now **echoes the typed `code`** in its body so the web can branch on it: `huntCatalogAlbum` reads `httpErrorCode()` and, on `ALBUM_NOT_IN_LIDARR` (a best-of/compilation absent from Lidarr — "The Best of Shaggy", "Grandes éxitos"), **auto-falls back to a raw network search** for `"<artist> <album>"` (`rawHuntFallback` → `executeSearch({ forceDirectOpen: true })`, a blue `data-testid="raw-fallback-note"` explains the switch) so the user still gets downloadable folder candidates from the network lane instead of a dead-end error. Placeholder release years (`0001`) are dropped at mapping time. The web's `huntCatalogAlbum` surfaces other (non-404) **server** error bodies via the shared `lib/http-error.ts` `httpErrorMessage()` helper — Angular's `HttpErrorResponse` is **not** an `instanceof Error`, so the naive `err instanceof Error ? err.message : …` swallowed the real reason (the 404 discography message, or a transient Lidarr/SkyHook metadata-API 503) and always showed a generic "Failed to prepare album". → See [docs/album-hunt.md](docs/album-hunt.md) and [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §A.
- **Album hunt**: `AlbumHunterService` fires base + skewed queries against slskd with soft-ban bypass (`buildSkewedQueries`), diacritic-insensitive scoring, two-phase endpoint for live UI progress, and a cross-peer fallback + auto-retry for exhausted jobs. When a hunt finds **0 folder candidates**, the modal offers **"Grab individual tracks instead"** → `POST /api/discography/albums/:id/hunt-tracks` resolves the Lidarr tracklist and runs `TrackHunterService` (one slskd search per title; best *cleanest* file picked by the shared pure `pickBestTrackFile` in `services/track-pick.ts`; grouped per peer + enqueued) — the §C1/§F2 per-track fallback for the album-only dead-end. → See [docs/album-hunt.md](docs/album-hunt.md).
- **Watchlist auto-hunt**: `WatchlistService` (`/api/watchlist`) lets users star catalog albums; a poller auto-hunts and downloads them when a confident candidate appears. → See [docs/album-hunt.md](docs/album-hunt.md).
- **Spotify metadata fallback (download via spotDL)**: a metadata-only lane that, when Soulseek comes up empty, finds the matching album in the Spotify catalog and downloads it through **spotDL** (which already handles `spotify.com` URLs) — so the candidate's `open.spotify.com/album/<id>` URL is handed straight to `POST /api/acquire` with no new acquire-path code. `SpotifySearchService` (OAuth client-credentials, cached app token, `/v1/search?type=album`) + `routes/spotify.ts` (`GET /api/spotify/search?q=` / `?artist=&album=`, gated on `plugins.isEnabled('spotify')` → 503) mirror the archive.org lane. The pure-JS **`spotify` plugin** (capability `search`, **no resolve/download**, default-off) holds the Spotify app **client id/secret** and gates the lane; the full flow needs the **spotdl** plugin too (the web gates one-click download on `hasSpotdl()` = enabled+available, else shows a manual "enable spotDL" note + external link). Surfaces as a "From Spotify" section in **both** the unified search page and the album-hunt modal. Credentials are entered in **Settings → Plugins** via a **new generic config-field editor**: a manifest may declare `configFields` (`{key,label,type:'text'|'password',…}`); `GET /api/plugins` echoes them + a `configured` map but **redacts `password` values**, a blank password input is dropped on save, and `registry.setConfig` **merges** over stored config so "leave the secret blank to keep it" round-trips (build/prefill logic in DI-free `lib/plugin-config.ts`). → See [docs/spotify-fallback.md](docs/spotify-fallback.md) and [docs/plugins.md](docs/plugins.md).
- **Idempotent hunt — one album = one download**: `hunt-download` guards with 409s for already-active or already-complete albums and enqueues **only missing tracks** via `filesMissingOnDisk`. → See [docs/album-hunt.md](docs/album-hunt.md).
- **Duplicate prevention**: Format preference (FLAC over MP3) + auto-dedupe after each organized batch. Album IDs are deterministically edition-collapsing so scanner-level dedup is inherent. → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Lossless → Opus standardization (storage + web playback)**: when `downloads.transcodeLossless.enabled`, lossless downloads (FLAC/WAV/…; `isLossless()` in `library-track-select.ts`) are transcoded to Opus **in place** by `post-download-transcode.ts` `transcodeToOpus()` — in `LibraryOrganizer.placeFile()` **after the move and before the scan**, so the song's path-derived id is computed once on the final `.opus` (no orphaned curation). Already-lossy files are left untouched; a transcode failure leaves the original in place. The **existing** library is migrated by `transcodeLibraryToOpus()` (`library-transcode.ts`) — `scripts/convert-library.ts` (dry-run unless `--apply`) or admin `POST /api/admin/transcode-library`. Re-encoding changes the path → `songId` → `acquisitions` key, so per file it carries `starred`/`hidden` to the new row and re-points `playlist_songs.song_id` + `acquisitions.relative_path` before dropping the stale lossless row (album-keyed artwork/release-meta survive on the unchanged `albumId`). Requires ffmpeg on PATH; gated by `ffmpegAvailable()`. → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Album deletion**: `DELETE /api/library/albums/:id` is folder-first (`rmSync` the whole `<Artist>/<Album>` dir), then synchronously deletes canonical rows — no tombstone/async-scan reconciliation needed. It also prunes the **now-orphaned aggregate rows** in the same transaction — the `library_artists` row (deleted if no releases/songs remain, else its `album_count` is corrected), the album's `library_artwork` row, and an emptied `library_genres` row — so deleting an artist's only release doesn't leave it lingering in search or as an empty artist page until the next *full* scan. → See [docs/download-pipeline.md](docs/download-pipeline.md) and [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §D.
- **Downloading albums suppressed from listing**: `GET /api/library/albums`, `GET /api/library/singles`, and `GET /api/library/artists/:id` exclude albums with an active `album_jobs` row (matched via `normalizeForGrouping`). The paginated endpoints (`/albums`, `/singles`) apply the exclusion **inside the SQL `WHERE` (`downloadingExclusion()` in `routes/library.ts`, an `id NOT IN (…)` fragment), not as a post-`LIMIT` JS filter** — filtering after `LIMIT/OFFSET` shrank each page below its requested size, so the web's offset-accumulating grid (`library.component.ts`) re-fetched already-shown rows and rendered the same album 2–3× (and tripped `done` early, truncating the grid). `downloadingExclusion()` has a zero-cost fast path when no downloads are active; the web also dedups defensively on append via `lib/append-unique.ts`. Albums appear the moment their job exits `active`. `DownloadWatcher.start()` seeds `knownCompleted` from `completed_downloads` at boot so container restarts don't replay history.
- **Untracked downloads**: Rows with `relative_path IS NULL` (pre-organizer) are backfilled by `backfillRelativePaths` (`scripts/backfill-untracked.ts`, dry-run unless `--apply`). Listed at `GET /api/library/untracked` (admin).
- **URL acquisition (yt-dlp / spotdl)**: `POST /api/acquire` routes the URL to an enabled `resolve`-capable **plugin** via `registry.getEnabledForUrl()` (no hardcoded backend detection) and runs downloads through the same `LibraryOrganizer` + incremental scan pipeline as Soulseek; 503 when no enabled plugin handles/can-serve the URL. Gated by plugin enable-state + binary on PATH. Completed/failed acquire jobs surface in the Downloads → Active tab ("URL Downloads" section) with Retry and Dismiss actions; done/failed rows older than 7 days are auto-pruned at startup. `AcquireJob` type lives in `@nicotind/core`. Uploads tab removed from the Downloads UI. The **archive.org** plugin is a third `resolve` backend on this same path — pasting an `archive.org` item URL routes to it. → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Download list metadata**: `GET /api/downloads` annotates in-flight folders matching `album_jobs` with `albumJob: { artistName, albumTitle, canonicalTrackCount }`. Web groups transfers via the pure `lib/download-groups.ts`.
- **Unified downloads feed**: the Downloads → Active tab is one feed — slskd groups + URL acquire jobs both adapt into a normalized `DownloadItem` (`lib/download-groups.ts`, `buildDownloadFeed`, stage-sorted) rendered by `components/download-item/` with a method badge (`lib/acquisition-method.ts`), a pipeline-stage chip (`components/pipeline-stage-badge/` + pure `lib/pipeline-stage.ts`), start time, and an expandable storage path; retry/cancel/remove are emitted and dispatched by `item.kind`. **A long, unbreakable title (e.g. a pasted Spotify URL) must truncate, not stretch the row** — three classes make that hold and are easy to drop: the host (the grid item) needs `block min-w-0` (an inline host sizes to content; a grid item won't shrink below its content's max width), the row needs `min-w-0 overflow-hidden`, and the title `<p>` (a flex child, so `min-width:auto` by default) needs `truncate min-w-0` or `truncate` is inert. The two fragile class lists are exported as `DOWNLOAD_ITEM_HOST_CLASS`/`DOWNLOAD_ITEM_TITLE_CLASS` and **bound** (not hardcoded) so a unit test asserts them — the JIT vitest harness can't drive the required `input()` into a render and jsdom can't measure layout, so the spec guards the class contract instead. → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Acquisition provenance (how/where-from/when)**: the `acquisitions` table (keyed on the final `relative_path`, the same join `library_songs.path` uses; side-table pattern like `library_artwork`) records `method` (`slskd`/`ytdlp`/`spotdl`/`archive`/`unknown`), `source_ref` (peer username or acquire URL), and timestamps. Written **at download time** — `download-watcher.ts` (slskd) and `acquire-watcher.ts` (URL) — via `acquisition-store.ts`, never reconstructed by fuzzy timing. `acquire_jobs` also gains `stage` (`PipelineStage`: queued→downloading→organizing→scanning→done/error) + `storage_path`. Types (`AcquisitionMethod`, `PipelineStage`, `SongAcquisition`) live in `@nicotind/core`. Surfaced per-track via `GET /api/library/songs/:id/acquisition` (404 unknown / `null` unrecorded) → an **"Acquisition" section** in the `track-info-sheet` drawer (badge from `lib/acquisition-method.ts`). Legacy songs are filled once at boot by `acquisition-backfill.ts` (joins `completed_downloads`, guarded by an `acquisitions_backfilled` `library_sync_state` marker). → See [docs/download-pipeline.md](docs/download-pipeline.md).
- **Plugin architecture (acquisition as opt-in plugins)**: acquisition (and, later, connectivity) is factored into opt-in plugins behind a kind-agnostic kernel. Contracts live in `@nicotind/core` (`src/plugin/*`: manifest + `Search/Browse/Resolve/Download/Connectivity` capability interfaces + `PluginHostContext` — the _only_ surface a plugin may use; plugins produce staged files and never touch the DB/organizer). `PluginRegistry` (`services/plugins/registry.ts`) registers build-time plugins, persists enable/consent/config (`plugins` + `plugin_kv` tables), and resolves plugins by kind/capability/URL. `routes/plugins.ts`: `GET /api/plugins` (any user, drives capability-gated UI), admin-only `POST :id/enable|disable` (consent-gated → `412` with disclaimer when required, records acting admin) + `PUT :id/config` (validated against the manifest's zod schema). **Compliance posture**: acquisition is default-off (zero capability until an admin enables a plugin). The **slskd plugin** (`services/plugins/slskd/`) wraps the Soulseek client and (de)registers its `SlskdSearchProvider` in `ProviderRegistry` on enable/disable — so network search/downloads/browse gate on it with no route changes; **hunt + watchlist** gate via `requireAcquisitionMiddleware` (`services/plugins/gate.ts`) + the poller's `isAcquisitionEnabled`. **yt-dlp + spotdl** are `resolve` plugins (`services/plugins/{ytdlp,spotdl}/`) on a shared process runner (`services/plugins/acquire/process.ts`); `AcquireWatcher` routes URLs via `registry.getEnabledForUrl()` (no more `detectBackend` enum) and owns job records + ingest, while plugins stage files. The **archive.org** plugin (`services/plugins/archive/`) is a third `resolve` backend but **pure JS** (no binary): it reads an item's `/metadata/<id>` file list, picks one audio format (MP3-preferred, FLAC fallback; `acquire.archive.preferredFormats`), and streams those files into staging. It also backs a **search lane** (`ArchiveSearchService` → `advancedsearch.php`, `GET /api/archive/search?q=` or `?artist=&album=`) gated specifically on the `archive` plugin (`plugins.isEnabled('archive')` → `503`, independent of slskd). Queries are **field-targeted + phrase-quoted** (`title:`/`creator:`) not bare full-text, **exclude the non-music audio collections** (`-collection:(librivoxaudio OR audiobooksandpoetry OR audio_bookspoetry OR oldtimeradio OR radioprograms OR radio OR podcasts OR audio_religion OR audio_news OR audio_tech OR gratefuldead OR etree)` — the broader `audiobooksandpoetry` umbrella + live-tape archives (`gratefuldead`/`etree`) flood "best of" queries, so a "Shaggy" search doesn't return a LibriVox audiobook), and are **`sort[]=downloads desc`** so popular real music outranks obscure mashups; results are then **deduped** by the pure `archiveDedupeKey()` (diacritic-folded, bracket/format/year noise stripped → sorted token set, so `Porfiado · El Cuarteto De Nos` and `El Cuarteto de Nos - Porfiado (2012) [FLAC]` collapse to one, popularity-sorted so the best copy survives). Each deduped item (capped at `MAX_ENRICH`) is then **enriched with a track count + `kind` (album/single)** via a bounded parallel `/metadata/<id>` lookup — the pure `countArchiveTracks()` takes the largest single-format audio group (so a FLAC+MP3 dual-encode counts once); items the lookup **proves have no audio files are dropped** (extra junk removal), a failed lookup leaves the count `null` (graceful). The web renders `creator · year · N tracks · album/single` via the pure `lib/archive-display.ts` `archiveSubtitle()`, omitting any absent piece (no literal "Unknown") so the user sees whether an item is a single or a multi-track album before acquiring. The service **retries once and throws `ServiceUnavailableError`** on an unreachable/non-OK upstream (route → `503`) so a transient archive.org failure shows as "unavailable" rather than a misleading empty result. → See [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §B. The web surfaces it in **two places** (both download by handing the item's `detailsUrl` to `/api/acquire`): the **album-hunt modal** (an "Also on archive.org" section run in parallel with the Soulseek hunt) and the **main unified search** ("From archive.org" section); both gate on `PluginService.hasArchive`. **Default-off compliance**: fresh installs enable nothing (`seedLegacyAcquisitionPlugins` migrates _existing_ installs once, guarded by an `app_settings` marker; archive is **not** seeded so it stays off for every install). **Web**: `PluginService` (signals, capability computeds `hasSearch`/`hasResolve`/`hasDownload` + `hasArchive`) + the admin **Settings → Plugins** page (`pages/plugins/`, route `/settings/plugins`); the search page gates the URL acquire box on `hasResolve` and the watchlist star on `hasDownload`. _Phases A–D landed; connectivity (E) is scaffolded (kind handled generically) but no connectivity plugin is shipped._ → See [docs/plugins.md](docs/plugins.md).

## Mobile app (Capacitor Android + iOS)

`packages/mobile` is a thin **Capacitor** shell that wraps the **same** `@nicotind/web` Angular build for Android — no second UI codebase (the Immich mentor ships native Flutter; we wrap to avoid double work for one maintainer, keeping only Immich's **server-URL entry screen** idea). The enabler is a **runtime-configurable API base URL**: the web historically assumed same-origin `/api/*`; now `ServerConfigService` (`apiUrl()`/`wsUrl()`, base `''` on web = unchanged, defaults to the canonical self-hosted server on native — platform detected via Capacitor's injected global in `lib/platform.ts`, **no `@capacitor/core` dep in the web bundle**) + the `authInterceptor` rewrite cover all HttpClient calls; direct URLs (stream/cover/Media-Session art in `player.component.ts`, the playback WS, `preserve.service.ts`, the admin log `EventSource`, share view) call `apiUrl()`/`wsUrl()`, and **cover `<img>`s rewrite centrally in `CoverArtComponent` (`resolvedSrc`)**. A **server-picker** (`pages/server-config/`, route `/server`, `serverGuard`) validates against `GET /api/health`; it's **native-only** (`needsConfiguration()` is always false on web, so the web/e2e are unaffected) and the service worker is disabled on native. The WebView is cross-origin, so the API mounts `nativeAppCors()` (`packages/api/src/middleware/cors.ts`) allowing the localhost Capacitor origins + `Authorization`/`Range` and **exposing the 206 range-streaming headers**. Android version is driven from `package.json` via `androidVersion()` (`packages/mobile/src/version.ts`) + `scripts/android-env.ts`; the CI `android` job (in `deploy.yml`, gated like `deploy`, **JDK 17**) builds the web, `cap sync`s, signs `assembleRelease` (keystore from secrets) and attaches the APK to the GitHub Release. **Background audio + lock-screen/notification controls** (the key native value-add) are handled by **`@jofr/capacitor-media-session`** — the Android WebView ignores the Web Media Session API and suspends backgrounded audio, so the plugin provides a native `MediaSession` + media-playback **foreground service**; it's wrapped by a lazily-imported `MediaControlsService` and fed by the pure `buildMediaMetadata` (`lib/media-metadata.ts`), with `player.component.ts` driving metadata/playback-state/handlers/position. The plugin officially targets **Capacitor 6**, so `packages/mobile` is pinned there (revisit on Cap 7+ support). The **native launcher/app icon AND the launch/splash screen are the NicotinD brand mark** (same SVG as the PWA manifest icon/favicon, not Capacitor's default bolt): one source in `src/native-icons.ts` (pure, unit-tested builders — `fullIconSvg`/`backgroundSvg`/`foregroundSvg` for the icon layers, `splashSvg` for the centred dark splash) → `scripts/generate-native-icons.ts` rasterizes the committed `assets/*.png` sources (icons 1024², splashes 2732²) via `sharp` → **`@capacitor/assets`** generates the Android mipmaps + splash drawables (committed) + iOS AppIcon + splash (generated in CI, since `ios/` is ephemeral). → See [docs/mobile-app.md](docs/mobile-app.md).

- **iOS (same Capacitor shell)**: the whole native-enablement layer above is **platform-neutral** and reused unchanged on iOS — `isNativePlatform()`, `ServerConfigService`/server-picker, `MediaControlsService` (the `@jofr` plugin is a Web Media Session wrapper on iOS), and `nativeAppCors()` (which **already allow-lists `capacitor://localhost`**, the iOS WKWebView origin — no API change). iOS-specific bits: `iosVersion()` (`packages/mobile/src/version.ts`, shares the monotonic scheme → `CFBundleShortVersionString`/`CFBundleVersion`) + `scripts/ios-env.ts`; an ephemerally-generated `ios/` project (Xcode/CocoaPods need a Mac, so it's **not committed** — CI runs `cap add ios` each run and patches `Info.plist` via `scripts/ios-plist.ts`/`src/ios-plist.ts` for `UIBackgroundModes: [audio]`); the macOS CI `ios` job (`deploy.yml`, gated like `deploy`) builds an **unsigned** IPA (`CODE_SIGNING_ALLOWED=NO`, no Apple account yet) and attaches it to the Release (install via AltStore/Sideloadly). **Background audio on iOS works**, but the `@jofr` plugin ships **no iOS native code** (its `dist/` only wraps WKWebView's Web Media Session API), and WKWebView wires play/pause to the `<audio>` element but does **not** surface JS-set metadata/artwork/position for cross-origin web audio — so the iOS lock-screen card showed controls but **no title/artist/artwork/time**. Fix: a minimal iOS-only Capacitor plugin **`@nicotind/capacitor-now-playing`** (`packages/capacitor-now-playing/`, Swift `NowPlayingPlugin` → `MPNowPlayingInfoCenter`) owns the *displayed info* (title/artist/album/artwork/duration/elapsed + `playbackState`); `MediaControlsService` routes `setMetadata`/`setPlaybackState`/`setPositionState` to it **only when `isIosNative()`** (via `getCapacitorPlugin('NicotindNowPlaying')` off the injected `Capacitor` global — no new web import; pure mapping `toNativeMetadata`/`pickArtworkUrl` in `lib/now-playing.ts`), falling back to `@jofr` if absent. **The plugin also owns transport — this is the fix for the card that never actually worked**: the original "just set `nowPlayingInfo`, let WKWebView keep transport, register no `MPRemoteCommandCenter`" design couldn't work, because iOS only *displays* `nowPlayingInfo` for the app that owns the now-playing session, which needs an **active `AVAudioSession` + ≥1 registered `MPRemoteCommandCenter` target** — both held by WKWebView, so the system read WebKit's empty session. So the plugin now activates `AVAudioSession(.playback)` (`ensureSession`, lazy on first play) and registers the lock-screen commands (`registerCommands`), forwarding each to JS via a single `remoteCommand` event. **Consequently iOS bypasses `@jofr` `setActionHandler`** — `MediaControlsService.setActionHandler` branches on `isIosNative()` and dispatches the native `remoteCommand` event to per-action handlers (wiring both would double-fire every press). The plugin is a workspace dep of `@nicotind/mobile` so `cap sync ios` discovers + pods it (no `deploy.yml` change); the Swift is build-gated by the macOS `ios` job. **No macOS CI test job** (cost) — the Swift is verified on-device via a `getDiagnostics()` plugin method surfaced in the **Settings → "Now Playing (iOS)"** panel (native-iOS-only, `data-testid="now-playing-diagnostics"`: plugin registered? session/category? commands registered? now-playing keys? last artwork status?), with `artworkError` events surfacing ATS/network artwork failures. Residual risk (device-only): activating our `AVAudioSession` could disturb the working background playback (mitigated: `.playback`, lazy activate, `setActive(false)` only on `clear()`). Two adjacent fixes shipped with this: (1) **safe-area header** — the sticky header sat *under* the iOS notch, fixed by `viewport-fit=cover` (`index.html`) + folding `env(safe-area-inset-top)` into the header's top padding (no-op on web); (2) **`MediaControlsService` `@jofr` proxy fix** — `session()` used to resolve its promise to the bare Capacitor `MediaSession` proxy, and Promise resolution probing `.then` on the proxy threw `MediaSession.then() is not implemented on web` (a real uncaught rejection in the production web console); it's now **boxed** (`{ session }`) so the proxy never lands on a resolution path. → See [docs/ios-app.md](docs/ios-app.md) and [docs/web-ui.md](docs/web-ui.md).

## End-to-end tests

`packages/e2e` is a Playwright suite that boots the real server (`bun run src/main.ts`) against a throwaway DB + committed silent-FLAC fixtures and drives the SPA in Chromium: auth, library, playback, **player controls (pause/next/seek/shuffle)**, and the compliance-critical plugin capability gating. No slskd/Lidarr needed — acquisition is default-off, so the test server runs `NICOTIND_MODE=external` with dead slskd/Lidarr URLs and a fresh `.tmp-data` dir per run (deterministic first-user admin). Selectors are `data-testid` attributes — **adding a `data-testid` is the standard for new e2e-targeted elements** (`app-password-field` forwards a `testId` input). Runs on port 8585 (override `E2E_PORT`); `E2E_BASE_URL` points it at a running instance (prod smoke) and skips the managed server. **CI is split across two workflows**: `ci.yml` ("CI & Release") runs the `ci` + `e2e` jobs on every push/PR (the `ci` job also **actionlint**s the workflow files themselves — a malformed `${{ }}` expression invalidates a whole workflow and fails it at startup with zero jobs, silently skipping deploy, so it's gated here), then a `release` job bumps the version and pushes a `vX.Y.Z` tag; that tag triggers `deploy.yml` ("Build & Deploy"), which SSH-deploys `git reset --hard <tag>` + `docker compose up`. The `android`/`ios` mobile-asset jobs in `deploy.yml` are **tag-only + `continue-on-error: true`** (best-effort Release artifacts — a flaky unsigned macOS build must not red-flag a successful server `deploy`). `deploy.yml` also accepts `workflow_dispatch` (manual redeploy of `origin/master`). A separate **gated playground harness** (`PLAYGROUND=1`, `tests/*.playground.ts`) drives real user flows against a live backend and **records observations** (timings, gaps, enhancement signals, cover-art 404s) into an aggregated findings report (`packages/e2e/playground-report/*.{md,json}`) via a custom reporter — the automated successor to the manual playground sessions. It degrades gracefully (never red on a dead backend) and stays out of CI; only its pure logic (`playground/{observe,report,net-monitor,console-monitor,journey}.ts`) is unit-tested in the `ci` job. **Each flow also captures runtime health and friction**: the fixture wires `page.on('console')`/`pageerror`/`requestfailed` → `console-monitor.ts` (a `console.error` during a normal flow is a defect signal even when every HTTP call was 200; a small ignore list drops the browser's generic "Failed to load resource" echo the response classifier already records), and `obs.journey()` (step/fallback/deadEnd, auto-flushed) + `obs.outcome(...)` feed a per-flow **Outcome matrix** + **Health summary** in the report. Gap-fill flows cover **downloads**, **playlists** (self-cleaning), **sharing** (anonymous read-only, self-cleaning) and **admin/plugins** (read-only; toggle gated by `PLAYGROUND_PLUGIN_TOGGLE`). The **real round-trip** (`tests/real-roundtrip.real.ts` via `playwright.real.config.ts`, `bun run … playground:real`) is the only mutating tier — it **acquires → verifies in-library → verifies playable → DELETEs** against prod with always-run cleanup, behind a two-key guard (`E2E_BASE_URL` **and** `PLAYGROUND_REAL=1`); never in CI. A parallel family of **screenshot flows** (`tests/*.screens.ts`, also out of CI) captures per-state mobile shots for UX review under `screenshots/mobile/<flow>/` (pure `playground/shot.ts` helper, CI-unit-tested): `playwright.live-screens.config.ts` (`bun run --filter @nicotind/e2e screens:live`) drives the live-backend **player-analysis** + **downloads-acquire** mobile journeys and — because they use the `obs` fixture — emits screenshots *and* the findings report in one run (mutating sub-steps gated behind `PLAYGROUND_ANALYZE` / `PLAYGROUND_ACQUIRE_URL`). **The flow catalogue (every main flow × which tier covers it) and the recurring routines (weekly live pass, pre-release real round-trip, screenshot pass, monthly log rotation) live in [docs/testing-routines.md](docs/testing-routines.md).** → See [docs/e2e.md](docs/e2e.md) "Playground harness" / "Real round-trip" / "Screenshot flows" and [docs/e2e-playground-findings-2026-06.md](docs/e2e-playground-findings-2026-06.md) §E2/§F.

**Real-use feedback log**: [docs/feedback-log-2026-06.md](docs/feedback-log-2026-06.md) is a rolling, dated log of friction noticed while actually *using* the app (distinct from the automated playground findings) — one entry per observation with Severity/Status, so recurring pain clusters surface as a prioritization signal ("is this tool useful?"). Rotate monthly.
